## 1. 核心概念

### 1.1 板级设备树（Board DTS）

- 每块板在 Zephyr 仓库中有 **`BOARD.dts`**（例如 `boards/st/nucleo_l4r5zi/nucleo_l4r5zi.dts`），由 SoC 的 **`*.dtsi`** 拼出片上外设、内存、`chosen` 等。
- **`chosen`** 中的 **`zephyr,sram`** 指向主 RAM 节点，用于生成 **`CONFIG_SRAM_*`** 与链接器 **`RAM`** 区域；**`zephyr,flash`** 指向 Flash。

### 1.2 Overlay（覆盖层）

- **Overlay** 是在板级 DTS **之上** 追加的片段，语法与 DTS 相同，用于在不改 Zephyr 上游板文件的前提下，覆盖或增删节点。
- 合并顺序由构建系统决定；后应用的 overlay 可覆盖先出现的属性（具体以 Zephyr 文档与实现为准）。

### 1.3 应用配置目录

- 默认 **`APPLICATION_CONFIG_DIR`** 等于**应用源码根目录**（含 `CMakeLists.txt`、`prj.conf` 的目录）。
- 若使用 Sysbuild 等修改了配置目录，以下路径均以实际配置目录为准。

## 2. Overlay 文件如何被构建系统发现

Zephyr 通过 CMake 模块 **`configuration_files.cmake`** 填充 **`DTC_OVERLAY_FILE`**。逻辑要点如下（未在命令行预先设置时）：

1. **`${APPLICATION_CONFIG_DIR}/socs/`**  
   按 SoC 限定名查找匹配的 overlay（带 `QUALIFIERS` 等）。

2. **`${APPLICATION_CONFIG_DIR}/boards/`**  
   按**当前板名**（及可选 SoC 限定）查找 **`<board>.overlay`** 或 **`shortened` 命名**（多 SoC 板时要求带 SoC 后缀的命名，见 Zephyr 报错提示）。

3. 若仍**未**得到 `DTC_OVERLAY_FILE`：  
   在 **`APPLICATION_CONFIG_DIR` 根目录** 再按板名规则查找 `*.overlay`。

4. 若仍没有：  
   再专门查找根目录下的 **`app.overlay`**（可配合 `FILE_SUFFIX` 等变体）。

**结论**：不是“放在工程里任意路径都会生效”，而是必须符合 **目录 + 命名 + 扩展名** 规则，或 **显式传入变量**（见下文）。

构建时可在日志中看到类似：

```text
-- Found devicetree overlay: D:/.../zephyr_template/app.overlay
```

## 3. 文件命名、扩展名与目录约定

### 3.1 自动发现只认 `.overlay`

- Zephyr 的 **`zephyr_file(CONF_FILES ... DTS ...)`** 对 DTS 类配置只收集 **`.overlay`** 后缀（见 `extensions.cmake` 中说明）。
- 若文件名为 **`boards/overlay.dts`**，**不会**自动作为 overlay 参与合并，除非通过 **`DTC_OVERLAY_FILE`** / **`EXTRA_DTC_OVERLAY_FILE`** 显式指定。

### 3.2 推荐做法

| 用途 | 推荐文件 |
|------|----------|
| 全工程默认、与板无关的通用覆盖 | 应用根目录 **`app.overlay`** |
| 仅针对某块板 | **`boards/<board>.overlay`** 或 SoC 子目录下按 Zephyr 规则命名 |
| 临时试验 | `west build ... -- -DDTC_OVERLAY_FILE=路径` |

### 3.3 `/delete-node/` 的语法位置

- **`/delete-node/ &label;`** 必须写在 **overlay 文件顶层**，**不能**放在 `/ { ... }` 节点内部，否则 Devicetree 解析会报错（`expected node name` 等）。

## 4. 手动指定 Overlay

### 4.1 `DTC_OVERLAY_FILE`

- 在 CMake 或 `west build` 中设置 **`DTC_OVERLAY_FILE`** 为**一个或多个** overlay 路径（分号或列表，按环境而定）。
- 一旦设置，**不再**走上一节的自动搜索逻辑（以 Zephyr 当前实现为准；若需叠加，见下条）。

### 4.2 `EXTRA_DTC_OVERLAY_FILE`

- 在自动或手动指定的 overlay **之后**再合并，适合“追加一层覆盖”而不改原 `DTC_OVERLAY_FILE` 列表。

### 4.3 与 Sysbuild

- 多镜像工程时，变量可能来自 Sysbuild 的 **全局/本地** 传递；若 overlay 未生效，检查 `APPLICATION_CONFIG_DIR` 与 `zephyr_get(... SYSBUILD ...)` 是否覆盖。

## 5. 主 RAM：`zephyr,sram` 与链接脚本

- **`chosen { zephyr,sram = &sram0; }`** 指向的 **`memory`** 节点，其 **`reg`** 的 `基址 + 长度` 决定：
  - Kconfig 中的 **`CONFIG_SRAM_BASE_ADDRESS`**、**`CONFIG_SRAM_SIZE`**（单位通常为 KB）；
  - 链接脚本中的 **`RAM`** 区域（`.data`、`.bss`、默认内核堆、多数栈等）。
- Zephyr 4.x 下链接脚本由 **CMake/模板生成**，**不要**沿用旧教程里手写 **`INCLUDE linker-base.ld`** 且路径相对于构建目录的方式，除非完全遵循当前 Zephyr 的 `HAVE_CUSTOM_LINKER_SCRIPT` 与模板要求。

## 6. 本工程：STM32L4R5ZI 与 `app.overlay`

### 6.1 SoC 默认 DTS 中的拆分

STM32L4R5 系列在 Zephyr DTS 中常将片上 SRAM 拆成多个 **`memory`** 节点（例如仅 `sram0` 作为 `zephyr,sram` 的 192KB，其余为 `zephyr,memory-region` 等）。**若只使用 192KB 主区**，在开启较大 **`CONFIG_HEAP_MEM_POOL_SIZE`** 与多线程栈时，**RAM 区域**可能链接失败（`region RAM overflowed`）。

### 6.2 本仓库的做法

在 **`app.overlay`** 中：

- 在**顶层** `/delete-node/` 删除 SoC 默认的 **`sram0`、`sram1`、`sram2`** 节点；
- 重新定义 **`sram0`**：`0x20000000` 起 **640 KiB（`0xA0000`）** 连续区域，与 RM0432 中 SRAM1+SRAM2+SRAM3 在地址空间上的连续映射一致；
- 板级 **`chosen`** 仍指向 **`&sram0`**，无需改 `nucleo_l4r5zi.dts`。

这样链接器中的 **`RAM`** 为 **640KB**，与 `west build` 末尾 **Memory region** 输出一致。

### 6.3 与 `prj_sram.conf` 的关系

- **`prj_sram.conf`** 为**说明性**片段（注释），可配合 **`prj.conf`** 使用；**Heap 大小**等仍由 **`CONFIG_HEAP_MEM_POOL_SIZE`** 等在 **`prj.conf`** 中配置。

## 7. 非连续或多块物理 RAM

### 7.1 设备树能描述什么

- 可在 DTS 中定义**多个** **`memory`** 节点，部分可带 **`compatible = "zephyr,memory-region"`** 与 **`zephyr,memory-region = "名字"`**，供 DMA、外设驱动或显式链接段使用。

### 7.2 Zephyr 默认不会“自动合并成一块堆”

- **`k_malloc`** 等使用的**默认内核堆**，通常来自 **`zephyr,sram`** 对应的一段**线性地址范围**（由 **`CONFIG_HEAP_MEM_POOL_SIZE`** 等在**该区域内**分配）。
- **设备树仅描述多块 RAM，并不会自动把散落物理块合成一个统一动态分配池**。

### 7.3 常见策略

| 场景 | 方向 |
|------|------|
| CPU 视角上**地址连续**（如本 L4R5 主 SRAM 区） | 合成一个 **`memory` 节点**（如本工程 `app.overlay`） |
| 多段**地址不连续** | 主镜像只选一块作 **`zephyr,sram`**；其余用 **multi-heap**、**mem_attr**、**链接脚本/重定位** 等分别管理（**示例见 [11.8](#example-scattered-memory)**） |
| 仅需某段给 DMA | 单独 **`memory-region`** + 驱动或应用固定缓冲区 |

---

## 8. 与 Kconfig 的配合

- **`CONFIG_HEAP_MEM_POOL_SIZE`**：内核堆大小，**不应超过**主 `RAM` 区域在扣除 `.bss`/栈等之后的实际可用空间（需结合 map 文件评估）。
- **`CONFIG_SRAM_SIZE`**、**`CONFIG_SRAM_BASE_ADDRESS`**：由 Devicetree 生成，**不要**在 `prj.conf` 里手工覆盖，除非明确知道后果。
- 其它栈大小（如 **`CONFIG_MAIN_STACK_SIZE`**、应用模块 Kconfig）与 **总 RAM** 一起影响是否溢出。

## 9. 验证与调试

1. **构建日志**  
   - 确认 **`Found devicetree overlay: ...`** 是否包含预期文件。  
   - 链接结束处的 **Memory region** 表：`RAM` 的 **Region Size** 是否为预期（如 640 KB）。

2. **生成树**  
   - 查看 **`build/zephyr/zephyr.dts`** 与 **`include/generated/zephyr/devicetree_generated.h`**，确认 `sram` 节点与 `reg`。

3. **map 文件**  
   - **`build/zephyr/zephyr.map`** 分析各段占用与 heap 位置。

4. **配置变更后**  
   - 若修改 overlay 或 `prj.conf`，建议使用 **`west build ... -p always`** 或清理后重配，避免 CMake 缓存导致旧树残留。

## 10. 常见问题

**Q1：`boards/overlay.dts` 写了但没生效？**  
自动发现只认 **`.overlay`**（除非用 **`DTC_OVERLAY_FILE`** 显式指定）。建议改名为 **`boards/<board>.overlay`** 或合并进 **`app.overlay`**。

**Q2：overlay 报 `parse error` / `expected node name`？**  
检查 **`/delete-node/`** 是否在**文件顶层**；检查是否误用了不支持的 **`#include`** 位置（视 Zephyr 版本与预处理流程而定）。

**Q3：仍报 `region RAM overflowed`？**  
确认 overlay 已被合并且 **`RAM` 区域大小已变大**；再减小 `CONFIG_HEAP_MEM_POOL_SIZE`、线程栈或关闭不需要的模块。

**Q4：能否用自定义 `.ld` 替代 overlay？**  
Zephyr 4.x 需启用 **`CONFIG_HAVE_CUSTOM_LINKER_SCRIPT=y`**，且脚本须符合当前内核模板；**优先**用 **Devicetree 描述 `zephyr,sram`**，与官方生成链路一致。

## 11. 配置示例

以下示例路径均以**应用根目录**（含 `CMakeLists.txt`、`prj.conf`）为当前目录；命令在 **PowerShell** 或 **bash** 中均可使用，注意路径分隔符按系统调整。

### 11.1 示例 A：扩展主 RAM（本仓库 `app.overlay` + `prj.conf`）

**场景**：Nucleo **STM32L4R5ZI** 默认 DTS 仅将约 **192KB** 作为 `zephyr,sram`，需与 RM0432 一致使用 **0x20000000 起 640KB** 连续 SRAM。

**`app.overlay`**（与仓库中一致，可整段复制）：

```dts
/* STM32L4R5ZI：zephyr,sram 扩展为 0x20000000 起 640KB 连续 SRAM（见 RM0432） */

/delete-node/ &sram0;
/delete-node/ &sram1;
/delete-node/ &sram2;

/ {
	sram0: memory@20000000 {
		/* 640 KiB = 0xA0000 */
		reg = <0x20000000 0xA0000>;
	};
};
```

**`prj.conf` 片段**（与主 RAM、堆、线程配合；数值可按需求改）：

```properties
# 内核堆（k_malloc；事件系统等会用到）
CONFIG_HEAP_MEM_POOL_SIZE=65536

# 主栈 / 空闲栈（按需）
CONFIG_MAIN_STACK_SIZE=1024
CONFIG_IDLE_STACK_SIZE=320
CONFIG_ISR_STACK_SIZE=2048

# 事件分发线程栈
CONFIG_EVENT_DISPATCHER_STACK_SIZE=2048

# Thread IPC 服务（若启用）
CONFIG_THREAD_IPC_SERVICE_STACK_SIZE=1280
```

**构建**：

```bash
west build -b nucleo_l4r5zi . -p always
```

**预期**：日志中出现 `Found devicetree overlay: .../app.overlay`，链接末尾 **Memory region** 中 **`RAM`** 的 **Region Size** 为 **640 KB**。

---

### 11.2 示例 B：仅针对某开发板——`boards/<board>.overlay`

**场景**：同一套源码在多块板上构建，只希望 **Nucleo L4R5** 扩展 SRAM，其它板不受影响。

在应用目录创建 **`boards/nucleo_l4r5zi.overlay`**，内容与示例 A 中 **`/delete-node/` + `sram0` 节点**相同。  
构建时指定该板：

```bash
west build -b nucleo_l4r5zi .
```

Zephyr 会在 **`boards/`** 下匹配 **`nucleo_l4r5zi.overlay`**。一旦该文件被找到并写入 **`DTC_OVERLAY_FILE`**，**通常不会再**走根目录 **`app.overlay`** 的自动回退逻辑，因此 **SRAM 扩展应二选一**：要么只用 **`boards/nucleo_l4r5zi.overlay`**，要么只用 **`app.overlay`**，避免两处重复定义同一节点。若必须同时使用两个文件，请用 **`DTC_OVERLAY_FILE`** 显式列出两个路径（见示例 C）。

### 11.3 示例 C：命令行指定 Overlay（临时试验）

**场景**：不移动文件，临时指定某 overlay；或合并多个文件。

**单个文件**（Windows 路径注意引号）：

```powershell
west build -b nucleo_l4r5zi . -- -DDTC_OVERLAY_FILE="D:/Code/3-Project/zephyr_template/my_ram.overlay"
```

**多个文件**（CMake 列表，分号分隔，整段加引号）：

```powershell
west build -b nucleo_l4r5zi . -- "-DDTC_OVERLAY_FILE=D:/proj/o1.overlay;D:/proj/o2.overlay"
```

> **注意**：一旦设置 **`DTC_OVERLAY_FILE`**，通常**不再**自动搜索 **`app.overlay`**；若仍需要 `app.overlay`，请把它一并写进 **`DTC_OVERLAY_FILE`** 列表。

### 11.4 示例 D：在自动 Overlay 之后追加——`EXTRA_DTC_OVERLAY_FILE`

**场景**：保留 **`app.overlay`** 的自动发现，再叠加一层只做小改动的文件（例如关某 UART、改 `status`）。

新建 **`extra.overlay`**：

```dts
/* 示例：关闭某节点（标签以板级 DTS 为准，需与 zephyr.dts 一致） */
&lpuart1 {
	status = "disabled";
};
```

**构建时追加**：

```powershell
west build -b nucleo_l4r5zi . -- -DEXTRA_DTC_OVERLAY_FILE="D:/Code/3-Project/zephyr_template/extra.overlay"
```

合并顺序：**先** `DTC_OVERLAY_FILE`（含自动或手动的 overlay），**后** `EXTRA_DTC_OVERLAY_FILE`。

### 11.5 示例 E：仅覆盖属性（不删除 `memory` 节点）

**场景**：不改 SRAM 布局，只调整 **`chosen`** 或外设状态。

```dts
/ {
	chosen {
		zephyr,console = &usart1;
	};
};

&usart1 {
	current-speed = <921600>;
};
```

保存为 **`app.overlay`** 或板级 **`boards/<board>.overlay`** 即可（无需 **`/delete-node/`**）。

---

### 11.6 示例 F：无法扩展主 RAM 时——用 Kconfig 省内存

**场景**：仍使用默认 **192KB** `zephyr,sram`，链接报 **`region RAM overflowed`**，且暂时不能改 DTS。

在 **`prj.conf`** 中酌情减小（数值仅为示例）：

```properties
CONFIG_HEAP_MEM_POOL_SIZE=32768
CONFIG_MAIN_STACK_SIZE=1024
CONFIG_SYSTEM_WORKQUEUE_STACK_SIZE=1024
CONFIG_EVENT_DISPATCHER_STACK_SIZE=1536
CONFIG_THREAD_IPC_SERVICE_STACK_SIZE=1024
```

并与 **[项目配置项说明.md](项目配置项说明.md)** 中模块相关项一并收紧；改后务必 **`west build ... -p always`** 并查看 **map** 确认余量。

---

### 11.7 示例 G：使用说明性片段 `prj_sram.conf`

本仓库 **`prj_sram.conf`** 为**注释型**说明文件，**不会**被 Zephyr 自动加载。若要把其中的 heap/栈建议应用到构建，需：

- **复制**所需行到 **`prj.conf`**，或  
- **合并构建**：  

```bash
west build -b nucleo_l4r5zi . -- -DCONF_FILE="prj.conf;prj_sram.conf"
```

（以 Zephyr 支持的 **`CONF_FILE`** 多文件合并语法为准；若某键在多个文件中重复，以后者或合并规则为准。）

### 11.8 示例 H：多段散落内存（地址不连续）

**场景**：芯片有两段（或多段）片上 SRAM，**物理地址不连续**（中间有洞或完全分离），例如：

- **段 A**：`0x20000000` 起 **128 KB**（主 RAM，放内核、默认堆、栈）  
- **段 B**：`0x24000000` 起 **64 KB**（独立 bank，仅给 DMA 缓冲、大缓冲区等）

**不能**像 L4R5 主 SRAM 那样在 CPU 视角合成一条连续 `reg`，也**不能**指望 `k_malloc` 自动跨两段分配。

#### 设备树（示意 overlay）

以下地址与大小为**虚构示意**，实际需按**数据手册**填写；节点名、标签需与板级 DTS 兼容（避免与已有 `&sram0` 冲突时可改用 `delete-node` 后重建）。

```dts
/* 示意：两段不连续 SRAM；仅段 A 作为 zephyr,sram */

/ {
	chosen {
		zephyr,sram = &sram_primary;
	};

	/* 主 RAM：链接器 RAM 区、CONFIG_SRAM_*、默认内核堆均基于此段 */
	sram_primary: memory@20000000 {
		reg = <0x20000000 0x20000>; /* 128 KiB，示例 */
	};

	/* 第二段：命名区域，供驱动/应用/显式链接使用，不是默认 k_malloc 池 */
	sram_aux: memory@24000000 {
		compatible = "zephyr,memory-region", "mmio-sram";
		zephyr,memory-region = "SRAM_AUX";
		reg = <0x24000000 0x10000>; /* 64 KiB，示例 */
	};
};
```

**要点**：

- **`chosen`** 里 **`zephyr,sram`** 只能指向**一段**（此处 `&sram_primary`）。  
- **`sram_aux`** 使用 **`compatible = "zephyr,memory-region", "mmio-sram"`** 与 **`zephyr,memory-region`** 字符串，便于在 Zephyr 里通过 **内存属性 / 保留区域** 或应用代码按区域使用。  
- 若 SoC 已有默认 `memory` 节点，可能需 **`/delete-node/ &sram0;`** 等再按上例重建，规则同前文 **`/delete-node/` 须置顶**。

#### 软件侧（与默认堆的关系）

| 需求 | 做法 |
|------|------|
| 默认 **`k_malloc`** / **`CONFIG_HEAP_MEM_POOL_SIZE`** | 仅占用 **`zephyr,sram`** 对应的一段；**不会**自动使用第二段。 |
| 第二段上做**独立小堆** | 见下文 **方式一：`k_heap` + `DT_REG_ADDR`/`DT_REG_SIZE`**。 |
| 多段统一走**属性选择**（如 external / non-cacheable） | 见下文 **方式二：Shared Multi-Heap**；或 SoC/板级已在启动时 `shared_multi_heap_add`。 |
| 编译期放进指定区（**静态缓冲**） | 见下文 **方式三：链接段**；或 **`k_heap` 把整个第二段当池**（方式一）。 |
| **DMA** | DTS 中 **`zephyr,memory-region-flags`**、Cache/MPU 与驱动约定；缓冲区优先从第二段分配。 |

以下 C 示例均假定 Overlay 里第二段节点 **`sram_aux`** 与上文 DTS 一致；若标签不同，请把 **`DT_NODELABEL(sram_aux)`** 改成你的 **`nodelabel`**。

---

#### 方式一：第二段上建独立 `k_heap`（推荐应用层起步）

思路：用 **`DT_REG_ADDR` / `DT_REG_SIZE`** 读出第二段物理地址与长度，**整段**交给 **`k_heap_init`**，之后用 **`k_heap_alloc` / `k_heap_free`**，与默认 **`k_malloc`** 互不干扰。

**`prj.conf`**：一般**无需**额外打开 `SHARED_MULTI_HEAP`（内核 **`k_heap`** 已随 **`CONFIG_HEAP`** 等可用，以你当前 Zephyr 的 `Kconfig` 为准）。

**示例代码**（放在任意 `.c`，于 `main` 或 `SYS_INIT` 中调用一次初始化）：

```c
#include <zephyr/kernel.h>
#include <zephyr/devicetree.h>
#include <zephyr/sys/util.h> /* UINT_TO_POINTER */

#define SRAM_AUX_NODE DT_NODELABEL(sram_aux)

static struct k_heap sram_aux_heap;

static int sram_aux_heap_init(void)
{
	if (!DT_NODE_EXISTS(SRAM_AUX_NODE)) {
		return -ENODEV;
	}

	void *mem = UINT_TO_POINTER(DT_REG_ADDR(SRAM_AUX_NODE));
	size_t bytes = DT_REG_SIZE(SRAM_AUX_NODE);

	k_heap_init(&sram_aux_heap, mem, bytes);
	return 0;
}

/* 启动阶段初始化；优先级按工程需要调整 */
SYS_INIT(sram_aux_heap_init, POST_KERNEL, CONFIG_KERNEL_INIT_PRIORITY_DEFAULT);

void *my_aux_alloc(size_t nbytes)
{
	return k_heap_alloc(&sram_aux_heap, nbytes, K_NO_WAIT);
}

void my_aux_free(void *ptr)
{
	k_heap_free(&sram_aux_heap, ptr);
}
```

**说明**：

- **`UINT_TO_POINTER`**：见 `<zephyr/sys/util.h>`，用于把 `uintptr_t` 转为可访问指针（若宏名随版本变化，可用 `(void *)DT_REG_ADDR(...)`，注意整数与指针宽度一致）。  
- **整段作堆池**：`k_heap` 会在该内存上维护元数据，**不要**再对该区间做其它用途的写入（除非缩小 `bytes` 留出保留区）。  
- **MPU / 总线**：第二段必须已被 CPU **可访问**；若上电后默认不可访问，需在 SoC/板级或 **`MPU`** 中配置，否则一访问就 Fault。  
- **对齐**：若该段用于 **DMA**，除分配对齐外，还需满足外设与 Cache 要求（如 **`cache_data_flush_range`** 等，依芯片而定）。

---

#### 方式二：Shared Multi-Heap（多区域、按“属性”选堆）

适用于希望与 Zephyr **`shared_multi_heap_*`** API 对齐、或由 SoC 已注册 **EXTERNAL / NON_CACHEABLE** 等区域时。需 **`CONFIG_SHARED_MULTI_HEAP=y`**（自动选中 **`CONFIG_MULTI_HEAP`**）。

**`prj.conf`**：

```properties
CONFIG_MULTI_HEAP=y
CONFIG_SHARED_MULTI_HEAP=y
```

**示例**（在板级未注册第二段时，由应用补充注册；若 **`shared_multi_heap_pool_init` 已由 SoC 调用过**，再次调用可能返回 **`-EALREADY`**，可忽略或只执行 **`shared_multi_heap_add`**）：

```c
#include <zephyr/kernel.h>
#include <zephyr/devicetree.h>
#include <zephyr/multi_heap/shared_multi_heap.h>

#define SRAM_AUX_NODE DT_NODELABEL(sram_aux)

static int smh_aux_register(void)
{
	struct shared_multi_heap_region reg = {
		/* 与业务约定：可用 enum 或自定义 uint32_t，需与 alloc 时一致 */
		.attr = SMH_REG_ATTR_EXTERNAL,
		.addr = DT_REG_ADDR(SRAM_AUX_NODE),
		.size = DT_REG_SIZE(SRAM_AUX_NODE),
	};
	int err;

	err = shared_multi_heap_pool_init();
	if (err != 0 && err != -EALREADY) {
		return err;
	}
	return shared_multi_heap_add(&reg, NULL);
}

SYS_INIT(smh_aux_register, POST_KERNEL, CONFIG_KERNEL_INIT_PRIORITY_DEFAULT);

void *dma_buf_alloc(size_t len)
{
	return shared_multi_heap_aligned_alloc(SMH_REG_ATTR_EXTERNAL, 32, len);
}
```

参考 Zephyr 树内示例：**`samples/boards/espressif/spiram_test`**（`shared_multi_heap_aligned_alloc` / `shared_multi_heap_free`）。不同 SoC 可能已在 **`soc_*` / `board_*`** 里注册外部 RAM，应用侧避免重复 `add` 同一段。

---

#### 方式三：链接段——把全局/静态对象放进命名区

若 DTS 中 **`zephyr,memory-region = "SRAM_AUX"`** 且 **`LINKER_DT_REGIONS()`** 已参与链接（取决于 SoC 与 Zephyr 版本），链接器会生成与 **`SRAM_AUX`** 对应的 **memory region** 与 **section**。可将只读/可写数据放入该段，例如（**section 名需与生成名一致**，常见为属性字符串的大写/下划线形式，以 **`build/zephyr/zephyr.map`** 为准）：

```c
/* 示例：大块静态缓冲放在 SRAM_AUX（名称以实际 map / devicetree_regions 为准） */
__attribute__((section(".SRAM_AUX")))
static uint8_t s_dma_static_pool[4096];
```

更稳妥的方式是使用 **`#include <zephyr/linker/devicetree_regions.h>`** 中的 **`LINKER_DT_NODE_REGION_NAME(DT_NODELABEL(sram_aux))`** 等宏生成与 DTS 一致的 region/section 名，并查阅当前 Zephyr 文档 **“Linker script generation / Devicetree memory regions”**。若链接报错找不到段名，说明该 SoC 未把 **`zephyr,memory-region`** 接进最终 **`linker.ld`**，可退回 **方式一** 仅用地址建 **`k_heap`**。

---

#### `prj.conf`（与方式一/二配合）

```properties
# 方式二需要：
# CONFIG_MULTI_HEAP=y
# CONFIG_SHARED_MULTI_HEAP=y

# 方式一通常仅需内核堆相关默认选项；若关闭默认 k_malloc 可单独评估
# CONFIG_HEAP_MEM_POOL_SIZE=...
```

---

#### 小结

**散落内存**的典型套路是：**一段做 `zephyr,sram` + 其它段做 `zephyr,memory-region`**。代码侧最常用的是 **`DT_REG_*` + `k_heap_*`**（方式一）；需要与 Zephyr 子系统统一的“按属性选区”时再用 **`shared_multi_heap_*`**（方式二）。**不要**在 DTS 里把不连续物理区“假合并”成一条连续 `reg`（除非硬件地址确实连续）。
