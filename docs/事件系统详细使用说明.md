## 概述

事件系统是 Zephyr RTOS 框架的核心组件，提供高性能、线程安全的发布 - 订阅（Publish-Subscribe）模式通信机制。该系统支持：

- **异步事件处理**：解耦事件生产者与消费者
- **优先级调度**：支持事件优先级分类
- **线程安全**：所有操作均经过互斥锁保护
- **动态订阅**：运行时动态注册/注销事件类型和订阅者
- **ISR 支持**：支持从中断上下文发布事件

### 主要特性

| 特性 | 描述 |
|------|------|
| 最大事件类型数 | 256 |
| 每类型最大订阅者 | 16 (可配置) |
| 事件队列大小 | 64 (可配置) |
| 调度器优先级 | 5 (可配置) |
| ISR 安全发布 | 支持 |

## 核心概念

### 事件（Event）

事件是系统中最基本的通信单元，包含以下字段：

```c
typedef struct {
    event_type_t    type;           // 事件类型标识符
    event_priority_t priority;      // 事件优先级
    uint32_t        timestamp;      // 事件创建时间戳（ticks）
    uint32_t        source_id;      // 源模块/组件 ID
    uint32_t        data_len;       // 事件数据长度
    void           *data;           // 事件数据指针
    bool            is_dynamic;     // 数据是否动态分配
} event_t;
```

### 事件类型（Event Type）

事件类型用于分类事件，系统支持最多 256 种事件类型。预定义的通用事件类型：

```c
#define EVENT_TYPE_GENERIC        1U    // 通用事件
#define EVENT_TYPE_SENSOR_DATA    10U   // 传感器数据事件
#define EVENT_TYPE_SENSOR_CONFIG  11U   // 传感器配置事件
```

### 订阅者（Subscriber）

订阅者是注册接收特定类型事件的回调函数：

```c
typedef struct {
    event_callback_t callback;      // 回调函数
    void            *user_data;     // 用户数据
    uint32_t         subscriber_id; // 唯一订阅者 ID
    bool             is_active;     // 是否激活
} subscriber_entry_t;
```

### 事件优先级

```c
typedef enum {
    EVENT_PRIORITY_LOW      = 10,   // 低优先级
    EVENT_PRIORITY_NORMAL   = 5,    // 普通优先级
    EVENT_PRIORITY_HIGH     = 2,    // 高优先级
    EVENT_PRIORITY_CRITICAL = 0     // 关键优先级
} event_priority_t;
```

### 组件说明

1. **事件队列（Event Queue）**
   - 基于 Zephyr `k_msgq` 实现
   - 支持优先级排队
   - 溢出时丢弃新事件

2. **事件调度器（Event Dispatcher）**
   - 独立线程运行
   - 从队列取出事件并通知订阅者
   - 可配置优先级和栈大小

3. **事件类型管理器（Event Type Manager）**
   - 维护事件类型注册表
   - 管理每个类型的订阅者列表
   - 线程安全的访问控制

## 内存与所有权

本节说明事件在 **入队、分发、回调** 各阶段由谁分配、谁释放，避免误用导致 **use-after-free、双重释放或内存泄漏**。

### 队列里到底存了什么

事件队列基于 Zephyr `k_msgq` 初始化，单条消息大小为 `sizeof(event_t)`：

- `k_msgq_put` 会把调用方传入指针所指的 **`event_t` 整结构体按字节拷贝**到队列环形缓冲区中，**不是**只存「指向 `event_t` 的指针」。
- 结构体里的 `data` 成员本身是 **指针**：拷贝的是 **指针的值**（地址），**不会**把 `data` 指向的缓冲区再复制一份。因此队列中的副本与发布时用的那份 `event_t` 可能 **指向同一块堆上的 payload**（若曾用 `event_create_with_data` 等分配过）。

分发线程 `k_msgq_get` 时，会把队列槽位里的 `event_t` **再拷贝一份到线程栈上的局部变量**，然后调用 `event_notify_subscribers`，最后在适当时机释放动态分配的 `data`（见下）。

### `event_publish_copy`（推荐）：模块无需手动释放

流程概要：

1. `event_create_with_data` 在堆上分配 **`event_t` 外壳** 与 **`data` 缓冲区**，并把调用方数据 `memcpy` 进缓冲区。
2. `k_msgq_put` 将 **`event_t` 的值** 写入队列；队列中的副本仍带着指向该缓冲区的 `data` 指针，且 `is_dynamic == true`。
3. 入队成功后，实现会把 **发布端手里那颗** `event_t` 上的 `data` 置为 `NULL`（避免随后 `event_free` 误释放仍被队列引用的 payload），再 **`k_free` 仅释放 `event_t` 外壳**。
4. 分发线程取出事件、执行回调后，若 `is_dynamic && data != NULL`，则 **`k_free(data)`**，完成 payload 生命周期。

**结论**：走 `event_publish_copy` 时，模块 **不需要** 也 **不应该** 再释放本次事件的 `data` 或 `event_t`；发布失败时由 `event_publish_copy` 内部统一 `event_free` 清理。

### `event_publish`：模块可能仍需负责「外壳」与生命周期

`event_publish` **只负责把当前 `event_t` 拷贝进队列**，**不会** 在发布返回后替你 `k_free` 你通过 `event_create` / `event_create_with_data` 得到的 **`event_t *` 指针本身**。

分发线程在处理完一条消息后 **仅会**：

- 对从 `k_msgq_get` 得到的 **栈上副本** 判断 `is_dynamic && data != NULL`，若为真则 **`k_free(event.data)`**。

**不会** 释放你当初 `k_malloc(sizeof(event_t))` 得到的那颗结构体，因为队列里从未保存「指向那颗结构体的指针」，只有 `event_t` 的 **值拷贝**。

因此分情形说明：

| 发布方式 | `event_t` 存放位置 | 动态 `data` | 谁释放 `data` | 谁释放 `event_t` 外壳 |
|----------|-------------------|-------------|---------------|-------------------------|
| `event_publish_copy` | 内部临时堆分配 | 有 | 分发线程 | `event_publish_copy` 入队成功后释放外壳 |
| `event_publish` + 栈上局部 `event_t` | 栈 | 无，或 `data` 指向调用方长期有效内存 | 若有 `is_dynamic` 堆数据则分发线程释放 `data` | 无需（栈自动结束） |
| `event_publish` + `event_create*` 堆对象 | 堆 | 有（`is_dynamic`） | 分发线程释放 `data` | **须由模块释放**，当前实现**不会**自动 `k_free` 该 `event_t *` → **易泄漏** |

**实践建议**：

- 需要堆上 payload 时，**优先使用 `event_publish_copy`**，避免遗漏释放 `event_t` 外壳。
- 若坚持使用 `event_publish(event_create_with_data(...))`，须在发布成功后 **自行释放 `event_t` 外壳**（且须在 **确认** 不会与队列中副本的 `data` 冲突的前提下设计——当前队列副本仍持有 `data` 指针，故 **只能** 释放 **外壳结构体本身**，不可在分发完成前 `event_free` 整颗事件，否则会 **use-after-free**）。因极易写错，**不推荐**该组合；请改用 `event_publish_copy`。

- **`event_publish` 失败**（如队列满）：队列未接管事件，须 **`event_free(event)`**（或等价释放），否则泄漏。

- **`event_publish` + 栈上 `event_t` 且 `data` 指向栈缓冲区**：`k_msgq_get` 可能晚于发布函数返回，回调执行时 **栈缓冲区可能已失效** → **禁止** 在发布函数返回后仍让队列里的 `data` 指向局部数组；此类场景请用 **`event_publish_copy`** 或堆上/静态缓冲。

### `event_publish_from_isr`

- ISR 中 **不应** 使用 `k_malloc` / `event_create` 等可能阻塞或与堆策略冲突的分配方式。
- 典型做法是使用 **静态** 或 **预分配** 的 `event_t`（或小对象池），仅填入 `type`、`priority` 等；若带 `data`，须保证 **`data` 指向的内存在分发线程读之前一直有效**（例如静态缓冲、环形缓冲写指针尚未覆盖的区域）。
- ISR 路径下 **不会** 由本实现替你释放「事件结构体」；动态 payload 的归属仍须与线程侧设计一致。

### 订阅回调中的规则

- 回调收到的 `const event_t *event` 指向 **分发线程栈上的副本**（或实现内部等价存储），**仅在回调执行期间有效**；**不要** 保存该指针供异步稍后使用（除非你们另有复制或引用计数约定）。
- **默认不要** 在回调里 `k_free((void *)event)` 或 `k_free(event->data)`：**动态 `data` 由分发线程在回调全部返回后释放**。若业务需要「接管」payload，必须在设计层明确所有权转移，并相应调整 `is_dynamic` 与释放点（当前公开实现 **未** 提供所有权转移 API，请勿自行假设）。

### 与「最佳实践」中示例的对应关系

文档其它处的 **「发布成功后不要访问 `event`」** 主要指：**不要** 再读写已交给 `event_publish` 的那块 **`event_t` 内存的内容**（队列里已是拷贝，改原结构无意义；若仍持有堆指针则涉及泄漏或双重释放策略）。**不等于**「系统会自动 `k_free` 你 `event_create` 返回的指针」——后者仅对 **`event_publish_copy` 内部** 成立。

## 使用指南

### 快速开始

#### 1. 初始化事件系统

```c
#include "event_system.h"

void app_main(void)
{
    // 初始化事件系统
    event_status_t ret = event_system_init();
    if (ret != EVENT_OK) {
        LOG_ERR("Failed to initialize event system: %d", ret);
        return;
    }

    // 启动事件调度器
    ret = event_system_start();
    if (ret != EVENT_OK) {
        LOG_ERR("Failed to start event system: %d", ret);
        return;
    }

    LOG_INF("Event system started successfully");
}
```

#### 2. 注册事件类型

```c
// 定义自定义事件类型
#define MY_EVENT_TYPE_SENSOR_READ    100U
#define MY_EVENT_TYPE_DATA_READY     101U
#define MY_EVENT_TYPE_ALARM          102U

void register_event_types(void)
{
    event_register_type(MY_EVENT_TYPE_SENSOR_READ, "sensor_read");
    event_register_type(MY_EVENT_TYPE_DATA_READY, "data_ready");
    event_register_type(MY_EVENT_TYPE_ALARM, "alarm");
}
```

#### 3. 订阅事件

```c
static uint32_t g_subscriber_id;

// 事件回调函数
static void my_event_callback(const event_t *event, void *user_data)
{
    LOG_INF("Received event: type=%d, priority=%d, timestamp=%u",
            event->type, event->priority, event->timestamp);

    if (event->data != NULL && event->data_len > 0) {
        // 处理事件数据
        uint8_t *data = (uint8_t *)event->data;
        // ... 处理数据 ...
    }

    // 访问用户数据
    int *my_data = (int *)user_data;
    LOG_INF("User data: %d", *my_data);
}

void subscribe_to_events(void)
{
    int user_data = 42;

    event_status_t ret = event_subscribe(
        MY_EVENT_TYPE_SENSOR_READ,
        my_event_callback,
        &user_data,  // 用户数据指针
        &g_subscriber_id
    );

    if (ret != EVENT_OK) {
        LOG_ERR("Failed to subscribe: %d", ret);
    }
}
```

#### 4. 发布事件

```c
// 方法 1: 发布静态事件（无动态数据）
void publish_simple_event(void)
{
    event_t event = {
        .type = MY_EVENT_TYPE_SENSOR_READ,
        .priority = EVENT_PRIORITY_NORMAL,
        .timestamp = k_uptime_get_32(),
        .source_id = 1,
        .data = NULL,
        .data_len = 0,
        .is_dynamic = false
    };

    event_publish(&event);
}

// 方法 2: 带 payload 时优先用 event_publish_copy（推荐）
// 调度器会释放堆上 payload；模块无需 event_free。
void publish_data_event(const void *data, size_t len)
{
    event_publish_copy(
        MY_EVENT_TYPE_DATA_READY,
        EVENT_PRIORITY_HIGH,
        data,
        len
    );
}

// 若必须用 event_create_with_data + event_publish：分发线程只会 k_free(data)，
// 不会 k_free(event_t 外壳) → 易泄漏；须在发布成功后自行 k_free(event) 仅外壳，
// 且不可调用 event_free(event)（会提前释放 data）。极易误用，请见「内存与所有权」。

// 方法 3: 使用便捷函数（自动复制数据）
void publish_copy_event(void)
{
    uint8_t sensor_data[4] = {0x01, 0x02, 0x03, 0x04};

    event_publish_copy(
        MY_EVENT_TYPE_SENSOR_READ,
        EVENT_PRIORITY_NORMAL,
        sensor_data,
        sizeof(sensor_data)
    );
}
```

### 完整示例

```c
#include <zephyr/kernel.h>
#include <zephyr/logging/log.h>
#include "event_system.h"

LOG_MODULE_REGISTER(event_example, LOG_LEVEL_INF);

// 定义事件类型
#define EVENT_TYPE_BUTTON    50U
#define EVENT_TYPE_LED       51U

// 全局订阅者 ID
static uint32_t button_sub_id;
static uint32_t led_sub_id;

// 按钮事件回调
static void button_callback(const event_t *event, void *user_data)
{
    LOG_INF("Button event received: data_len=%u", event->data_len);
    
    if (event->data_len == sizeof(uint8_t)) {
        uint8_t button_state = *(uint8_t *)event->data;
        LOG_INF("Button state: %s", button_state ? "PRESSED" : "RELEASED");
    }
}

// LED 事件回调
static void led_callback(const event_t *event, void *user_data)
{
    LOG_INF("LED event received");
    
    if (event->data_len == sizeof(uint8_t)) {
        uint8_t led_state = *(uint8_t *)event->data;
        // 控制 LED
        LOG_INF("LED turned %s", led_state ? "ON" : "OFF");
    }
}

// 按钮线程（发布按钮事件）
static void button_thread(void *p1, void *p2, void *p3)
{
    while (1) {
        uint8_t button_state = 1;  // 模拟按钮按下
        
        event_publish_copy(
            EVENT_TYPE_BUTTON,
            EVENT_PRIORITY_NORMAL,
            &button_state,
            sizeof(button_state)
        );

        k_msleep(1000);
        
        button_state = 0;  // 模拟按钮释放
        event_publish_copy(
            EVENT_TYPE_BUTTON,
            EVENT_PRIORITY_NORMAL,
            &button_state,
            sizeof(button_state)
        );

        k_msleep(1000);
    }
}

// LED 控制线程（发布 LED 事件）
static void led_thread(void *p1, void *p2, void *p3)
{
    k_sleep(K_SECONDS(2));  // 等待按钮线程先启动

    while (1) {
        uint8_t led_state = 1;  // LED ON
        
        event_publish_copy(
            EVENT_TYPE_LED,
            EVENT_PRIORITY_LOW,
            &led_state,
            sizeof(led_state)
        );

        k_msleep(500);
        
        led_state = 0;  // LED OFF
        event_publish_copy(
            EVENT_TYPE_LED,
            EVENT_PRIORITY_LOW,
            &led_state,
            sizeof(led_state)
        );

        k_msleep(500);
    }
}

// 主初始化函数
void event_system_example_init(void)
{
    // 1. 初始化事件系统
    event_system_init();

    // 2. 注册事件类型
    event_register_type(EVENT_TYPE_BUTTON, "button");
    event_register_type(EVENT_TYPE_LED, "led");

    // 3. 订阅事件
    event_subscribe(EVENT_TYPE_BUTTON, button_callback, NULL, &button_sub_id);
    event_subscribe(EVENT_TYPE_LED, led_callback, NULL, &led_sub_id);

    // 4. 启动事件调度器
    event_system_start();

    // 5. 创建工作线程
    K_THREAD_DEFINE(button_tid, 1024, button_thread, NULL, NULL, NULL,
                    5, 0, K_NO_WAIT);
    K_THREAD_DEFINE(led_tid, 1024, led_thread, NULL, NULL, NULL,
                    6, 0, K_NO_WAIT);

    LOG_INF("Event system example initialized");
}
```

### 3. 内存管理

完整约定见上文 **[内存与所有权](#内存与所有权)**。此处仅列常用写法。

```c
// ✅ 推荐：event_publish_copy — 堆上 payload 与临时 event_t 外壳均由实现按序释放
void safe_publish(void)
{
    uint8_t data[16];
    event_publish_copy(EVENT_TYPE_GENERIC, EVENT_PRIORITY_NORMAL,
                       data, sizeof(data));
}

// ✅ 可行：栈上 event_t、无动态 data（或 data 指向全局/静态长期有效内存）
void stack_publish(void)
{
    static uint32_t counter;
    event_t event = {
        .type = EVENT_TYPE_GENERIC,
        .priority = EVENT_PRIORITY_NORMAL,
        .timestamp = k_uptime_get_32(),
        .data = &counter,
        .data_len = sizeof(counter),
        .is_dynamic = false,
    };
    event_publish(&event);
}

// ⚠️ 若误用 event_create_with_data + event_publish：至少保证失败路径释放整颗事件
void manual_publish_on_failure_only(void *data, size_t data_len)
{
    event_t *event = event_create_with_data(
        EVENT_TYPE_GENERIC, EVENT_PRIORITY_NORMAL, data, data_len);
    if (event == NULL) {
        return;
    }
    if (event_publish(event) != EVENT_OK) {
        event_free(event);  /* 未入队，整颗事件仍归调用方 */
    }
    /* 发布成功时：data 由分发线程释放，但 event_t 外壳不会自动释放 → 应改用
     * event_publish_copy，勿在生产中依赖本组合。 */
}
```

### 4. 错误处理

```c
void robust_publish(const void *data, size_t len)
{
    event_status_t ret = event_publish_copy(
        EVENT_TYPE_SENSOR_DATA,
        EVENT_PRIORITY_NORMAL,
        data,
        len
    );

    switch (ret) {
    case EVENT_OK:
        // 成功
        break;
    case EVENT_ERR_QUEUE_FULL:
        LOG_WRN("Event queue full, data dropped");
        break;
    case EVENT_ERR_INVALID_ARG:
        LOG_ERR("Invalid event arguments");
        break;
    case EVENT_ERR_NO_MEM:
        LOG_ERR("Memory allocation failed");
        break;
    default:
        LOG_ERR("Unknown error: %d", ret);
        break;
    }
}
```

## 故障排除

### 常见问题

#### 1. 事件未送达

**症状**：发布事件后回调未执行

**检查清单**：
- [ ] 事件类型是否已注册？
- [ ] 订阅者是否成功订阅？
- [ ] 事件调度器是否已启动？
- [ ] 回调函数是否正确实现？

```c
// 调试步骤
void debug_event_delivery(void)
{
    // 检查事件类型
    LOG_INF("Event type name: %s", event_get_type_name(MY_EVENT_TYPE));

    // 检查订阅者数量
    uint32_t count = event_get_subscriber_count(MY_EVENT_TYPE);
    LOG_INF("Subscriber count: %u", count);

    // 检查系统状态
    LOG_INF("System running: %s", 
            event_system_is_running() ? "yes" : "no");
}
```

#### 2. 队列溢出

**症状**：日志显示 "Event queue full, event dropped"

**解决方案**：
```kconfig
# 增加队列大小
CONFIG_EVENT_QUEUE_SIZE=128

# 或优化事件处理速度
# 减少回调中的处理时间
```

#### 3. 内存泄漏 / 野指针

**症状**：系统运行一段时间后内存不足，或随机崩溃

**检查点**：
- 优先使用 `event_publish_copy`；详见 [内存与所有权](#内存与所有权)
- `event_publish` + `event_create*`：分发线程**不会**释放 `event_t` 外壳，堆路径易泄漏；失败路径须 `event_free`
- **禁止** `event_publish` 的 `data` 指向发布函数内的局部栈缓冲（回调可能晚于函数返回）
- ISR 中避免堆分配；`data` 指向的内存在分发前须一直有效
- 回调内不要 `k_free(event->data)`（除非项目另有所有权约定）

#### 4. 死锁问题

**症状**：系统卡死，看门狗复位

**原因**：在事件回调中调用可能阻塞的 API

**解决方案**：
```c
// ❌ 错误示例
static void bad_callback(const event_t *event, void *user_data)
{
    k_mutex_lock(&some_mutex, K_FOREVER);  // 可能死锁
}

// ✅ 正确示例
static void good_callback(const event_t *event, void *user_data)
{
    k_sem_give(&processing_sem);  // 非阻塞
}
```

### 错误码说明

| 错误码 | 值 | 描述 |
|--------|-----|------|
| EVENT_OK | 0 | 成功 |
| EVENT_ERR_NO_MEM | -1 | 内存不足 |
| EVENT_ERR_QUEUE_FULL | -2 | 队列已满 |
| EVENT_ERR_QUEUE_EMPTY | -3 | 队列为空 |
| EVENT_ERR_INVALID_ARG | -4 | 无效参数 |
| EVENT_ERR_NOT_FOUND | -5 | 未找到 |
| EVENT_ERR_NO_SUBSCRIBER | -6 | 无订阅者 |
| EVENT_ERR_TIMEOUT | -7 | 超时 |
