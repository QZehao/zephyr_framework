# 项目配置项说明

本文档整理本仓库 **应用侧 Kconfig**（根目录 `Kconfig` 与 `src/modules/ipc_service/Kconfig`）中各配置项的含义。构建后由 Zephyr 生成 `autoconf.h`，源码中通过 `CONFIG_<符号名>` 使用（如 `CONFIG_EVENT_SYSTEM`）。

- **设置方式**：`prj.conf` 中写 `CONFIG_xxx=y` 或 `CONFIG_xxx=数值`，或使用 `west build -t menuconfig` / `guiconfig`。
- **符号前缀**：下文「配置宏」列即 C 代码中的宏名（与 `CONFIG_` 一致）。
- **与板级/内核选项**：SOC、驱动、内核 heap 等见 Zephyr 与板级 `defconfig`；本表仅覆盖**本应用扩展**的菜单项。

## 1. 事件系统（Event System Configuration）

| 配置宏 | 类型 | 默认值 | 含义 |
|--------|------|--------|------|
| `CONFIG_EVENT_SYSTEM` | bool | y | 是否启用事件系统：发布/订阅、事件队列与分发线程。关闭后依赖事件系统的功能不可用。 |
| `CONFIG_EVENT_QUEUE_SIZE` | int | 64 | 全局事件队列可容纳的最大事件个数（范围 16–512）。过大占 RAM，过小易丢事件。 |
| `CONFIG_EVENT_MAX_SUBSCRIBERS` | int | 16 | **每种事件类型**下允许的最大订阅者数量（范围 4–64）。 |
| `CONFIG_EVENT_DISPATCHER_STACK_SIZE` | int | 2048 | 事件分发线程栈大小（字节，范围 512–8192）。回调较深或局部变量大时需适当加大。 |
| `CONFIG_EVENT_DISPATCHER_PRIORITY` | int | 5 | 分发线程优先级（范围 1–15，**数值越小优先级越高**）。 |

**说明**：事件类型注册、订阅 API 等见 [事件系统详细使用说明.md](事件系统详细使用说明.md)。

## 2. 模块管理器（Module Manager Configuration）

| 配置宏 | 类型 | 默认值 | 依赖 | 含义 |
|--------|------|--------|------|------|
| `CONFIG_MODULE_MANAGER` | bool | y | — | 是否启用模块管理器：注册/注销、`init/start/stop` 生命周期、与事件订阅集成。 |
| `CONFIG_MAX_MODULES` | int | 16 | `MODULE_MANAGER` | 可同时注册的业务模块**槽位**上限（范围 4–32）。 |
| `CONFIG_MODULE_INIT_TIMEOUT_MS` | int | 1000 | `MODULE_MANAGER` | 单次 `register()` 中调用模块 `init()` 的允许最大耗时（毫秒，范围 100–10000）。超时则注册失败；为 0 表示不检查（以代码为准，见 `module_manager.h`）。 |
| `CONFIG_MODULE_MANAGER_RUNTIME_DEPENDENCIES` | bool | n | `MODULE_MANAGER` | 启用后：`module_manager_start_all()` 按各模块 `depends_on`（模块名字符串数组，NULL 结尾）做拓扑排序；`module_manager_stop_all()` 在 RUNNING 集合上按依赖逆序停止。非法依赖会经不动点剔除；成环或异常时回退为仅按 `priority` 排序。 |
| `CONFIG_MODULE_MANAGER_DEPENDS_LIST_MAX` | int | 16 | `MODULE_MANAGER_RUNTIME_DEPENDENCIES` | **单个模块**的 `depends_on` 中最多扫描多少个依赖名（不含末尾 NULL），防止未正确 NULL 终止时死循环。**不是**全系统模块总数，**不是**依赖链深度。范围 4–64。 |
| `CONFIG_MODULE_MANAGER_START_ALL_ABORT_ON_FAILURE` | bool | n | `MODULE_MANAGER` | 为 y 时：`module_manager_start_all()` 在首次 `start` 返回非 0 后不再启动后续模块。使用运行时依赖时建议开启，避免依赖方在依赖失败仍被启动。 |
| `CONFIG_EXAMPLE_MODULE_MULTI_DEP` | bool | n | `MODULE_MANAGER` | 编译并注册示例 `example_module_multi_dep`（`depends_on` 指向 `example_module_a` / `example_module_b`）。若需观察拓扑序，需同时开启 `RUNTIME_DEPENDENCIES` 并注册 A/B。 |
| `CONFIG_EXAMPLE_MODULE_GPIO` | bool | n | `MODULE_MANAGER` | 编译并注册 `example_module_gpio`（需板级 DTS 提供 **`led0`**；可选 **`sw0`** 用户键）。 |
| `CONFIG_EXAMPLE_MODULE_UART` | bool | n | `MODULE_MANAGER` | 编译并注册 `example_module_uart`（默认绑定 **`zephyr_console`** 对应 UART，可能与 Shell 共用）。 |
| `CONFIG_EXAMPLE_MODULE_UART_USE_ZEPHYR_CONSOLE` | bool | y | `EXAMPLE_MODULE_UART` | 为 y 时使用 `DEVICE_DT_GET(DT_CHOSEN(zephyr_console))`；为 n 时使用 `device_get_binding(CONFIG_EXAMPLE_MODULE_UART_DEVICE_NAME)`。 |
| `CONFIG_EXAMPLE_MODULE_UART_DEVICE_NAME` | string | `UART_0` | `EXAMPLE_MODULE_UART` | 上一项为 n 时的设备名字符串。 |

**说明**：`module_interface_t.depends_on`、`DECLARE_MODULE_*` 宏与示例见 [模块系统详细使用说明.md](模块系统详细使用说明.md)。GPIO/UART 叠加配置示例见仓库根目录 **`prj_example_gpio_uart.conf`**。

## 3. 系统服务（System Services Configuration）

各 **`sys_*`** 模块的职责与使用注意见 **[系统服务使用说明.md](系统服务使用说明.md)**。下列选项供应用侧服务（日志、内存池、看门狗等）读取；具体是否实现以源码为准。

**`sys_log` 目的地**：由 `sys_log_config_t.destinations`（**`sys_log_dest_mask_t`**）按位选择 `SYS_LOG_DEST_CONSOLE`、`SYS_LOG_DEST_MEMORY`、`SYS_LOG_DEST_RTT`（需 **`CONFIG_SEGGER_RTT`**）、`SYS_LOG_DEST_UART`；掩码为 0 时初始化会回退为「控制台 + 内存环」。

| 配置宏 | 类型 | 默认值 | 含义 |
|--------|------|--------|------|
| `CONFIG_SYS_LOG_LEVEL` | int | 3 | 系统/应用日志级别：0=关闭，1=ERROR，2=WARNING，3=INFO，4=DEBUG。 |
| `CONFIG_SYS_MEMORY_POOL_SIZE` | int | 8192 | 系统内存池大小（字节，范围 1024–65536），用于事件等分配场景时需与 `CONFIG_HEAP_MEM_POOL_SIZE` 等配合。 |
| `CONFIG_SYS_WATCHDOG_ENABLE` | bool | y | 是否启用看门狗相关逻辑（具体喂狗实现见 `sys_watchdog` 等）。 |
| `CONFIG_SYS_WATCHDOG_TIMEOUT_MS` | int | 5000 | 看门狗超时时间（毫秒，范围 1000–30000）。 |

## 4. Thread IPC 服务（in-app）

对应菜单 **「Thread IPC service (in-app)」**，定义见 `src/modules/ipc_service/Kconfig`。注意：此为**应用内双线程 + 队列**的 IPC，**不是** Zephyr 上游子系统 `IPC_SERVICE`（RPMSG/icmsg）。

| 配置宏 | 类型 | 默认值 | 依赖 | 含义 |
|--------|------|--------|------|------|
| `CONFIG_THREAD_IPC_SERVICE` | bool | n | — | 总开关：启用 Thread IPC 服务（工作线程 + 分发线程、请求/响应队列等）。 |
| `CONFIG_THREAD_IPC_SERVICE_MAX_PENDING_REQUESTS` | int | 16 | `THREAD_IPC_SERVICE` | 可同时跟踪的未完成请求数量上限（范围 4–64）。 |
| `CONFIG_THREAD_IPC_SERVICE_REQUEST_QUEUE_SIZE` | int | 8 | `THREAD_IPC_SERVICE` | 工作线程侧请求队列深度（静态缓冲，范围 2–32）。 |
| `CONFIG_THREAD_IPC_SERVICE_RESPONSE_QUEUE_SIZE` | int | 8 | `THREAD_IPC_SERVICE` | 分发线程侧响应队列深度（范围 2–32）。 |
| `CONFIG_THREAD_IPC_SERVICE_STACK_SIZE` | int | 2048 | `THREAD_IPC_SERVICE` | **每个**内部线程（worker / dispatcher）的栈大小（字节，范围 512–8192）。 |
| `CONFIG_THREAD_IPC_SERVICE_PRIORITY` | int | 5 | `THREAD_IPC_SERVICE` | 两线程优先级（范围 -16–15，依 Zephyr 约定）。 |
| `CONFIG_THREAD_IPC_SERVICE_LOG_LEVEL` | int | 3 | `THREAD_IPC_SERVICE` | 该模块日志级别：0–4，同上。 |
| `CONFIG_THREAD_IPC_SERVICE_EXAMPLE` | bool | n | `THREAD_IPC_SERVICE` | 是否编译内置 SYS_INIT 演示代码（与 `example_module_ipc` 二选一可省 RAM）。 |
| `CONFIG_THREAD_IPC_SERVICE_EVENT_BRIDGE` | bool | n | `THREAD_IPC_SERVICE` + `EVENT_SYSTEM` | 启用后将 IPC 结果桥接到事件系统（`ipc_service_event`）。 |
| `CONFIG_EXAMPLE_MODULE_THREAD_IPC` | bool | n | `THREAD_IPC_SERVICE` + `MODULE_MANAGER` | 编译 `example_module_ipc`：在 `module_manager` 中注册并与 Thread IPC 集成。 |

**说明**：架构与 API 见 **[Thread_IPC服务使用说明.md](Thread_IPC服务使用说明.md)**、**[Thread_IPC模块集成指南.md](Thread_IPC模块集成指南.md)**。

## 5. 应用头文件中的功能开关（非 Kconfig）

`src/app/app_config.h` 中的 **`APP_CONFIG_*`** 等为**编译期宏**，不在 `menuconfig` 中，常见项如下（以文件为准）：

| 宏 | 含义 |
|----|------|
| `APP_CONFIG_ENABLE_MODULE_A` / `B` | 是否在 `app_register_modules()` 中注册示例模块 A/B。 |
| `APP_CONFIG_ENABLE_LOGGING` 等 | 是否启用日志、看门狗、内存管理器、定时器服务等应用层特性。 |
| `APP_CONFIG_ENABLE_STATS` | 为 `1` 时：事件分发器启用运行时统计，Shell **`app events`** 额外打印 `event_dispatcher_get_stats()`；为 `0` 时仅事件系统全局统计。 |
| `APP_CONFIG_ENABLE_LOG_DUMP` | 为 `1` 时 Shell **`app log`** 调用 `sys_log_dump()`；为 `0` 时提示已禁用（仍可通过 API 使用）。 |

运行时依赖排序与上述开关独立；多依赖示例需 Kconfig 打开 `EXAMPLE_MODULE_MULTI_DEP` 且通常保留 A/B 注册。

## 6. 常用 Zephyr 选项（与本应用配合）

下列**不在**本仓库 `Kconfig` 菜单中，但 `prj.conf` 中经常需要修改：

| 配置宏 | 含义 |
|--------|------|
| `CONFIG_HEAP_MEM_POOL_SIZE` | 内核堆大小。事件系统 `event_create` / `event_publish_copy` 等使用 `k_malloc` 时需**非 0**。 |
| `CONFIG_MAIN_STACK_SIZE` | 主线程栈，业务较重时需加大。 |

板级外设、网络、文件系统等请参考对应 Zephyr/Kconfig 文档。
