## 发布前准备

### 代码质量

- [ ] **运行所有单元测试并通过**

  ```bash
  # 运行 native_posix 单元测试
  west build -b native_posix tests/ --build-dir build_tests
  west build -t run --build-dir build_tests
  ```

- [ ] **运行 clang-format 格式化代码**

  ```bash
  # Linux/macOS
  find src -name "*.c" -o -name "*.h" | xargs clang-format -i

  # Windows PowerShell
  Get-ChildItem -Recurse -Include *.c,*.h src/ | ForEach-Object { clang-format -i $_.FullName }

  # Windows CMD
  for /r src %f in (*.c *.h) do @clang-format -i "%f"

  # 或使用 pre-commit（推荐，跨平台一致）
  pre-commit run --all-files
  ```

- [ ] **运行 clang-tidy 静态分析**

  ```bash
  # 运行静态分析（如有 .clang-tidy 配置）
  run-clang-tidy src/**/*.c
  ```

- [ ] **确认无编译警告**

  ```bash
  # 清理并重新构建，检查警告
  west build -t pristine -b <your_board> .
  # 检查输出中无 warning
  ```

- [ ] **更新 CHANGELOG.md**

  在 `CHANGELOG.md` 顶部添加新版本条目：

  ```markdown
  ## [1.1.0] - 2026-04-01

  ### Added
  - 新增板型迁移指南文档
  - 添加设备树迁移详细步骤

  ### Changed
  - 优化内存配置流程
  - 更新文档索引

  ### Fixed
  - 修复小 RAM 板型栈溢出问题
  - 修正 overlay 说明文档

  ### Removed
  - 移除冗余的 boards/app.overlay
  ```

### 文档

- [ ] **更新 README.md（如有必要）**
  - 更新特性列表
  - 更新版本号（`bump_version.py` 会自动更新）
  - 更新快速开始示例（如有变更）

- [ ] **生成 API 文档**

  ```bash
  # Windows (PowerShell)
  .\scripts\generate_docs.ps1

  # Linux/macOS
  ./scripts/generate_docs.sh

  # 或手动运行 Doxygen
  doxygen Doxyfile
  ```

- [ ] **更新相关文档**
  - `docs/环境搭建与配置指南.md`（如有配置变更）
  - `docs/板型迁移指南.md`（如有新板型支持）
  - `docs/常见问题与故障排除.md`（如解决新问题）

- [ ] **确认文档与代码一致**
  - 检查 API 文档与实际代码一致
  - 检查示例代码可运行
  - 检查截图与当前 UI一致

### 版本管理

- [ ] **确认当前版本号**

  ```bash
  # 查看当前版本
  python scripts/bump_version.py
  # 或查看 APP_VERSION 文件
  cat APP_VERSION
  ```

- [ ] **确定新版本号**（遵循语义化版本规范）
  - **主版本号**：不兼容的 API 变更
  - **次版本号**：向后兼容的功能新增
  - **修订号**：向后兼容的问题修正

- [ ] **更新版本号**

  ```bash
  # 更新版本号（自动同步到 APP_VERSION、Doxyfile、README.md）
  python scripts/bump_version.py 1.1.0
  ```

- [ ] **提交版本变更**

  ```bash
  git add APP_VERSION Doxyfile README.md
  git commit -m "chore: 更新版本号至 1.1.0"
  ```

### CI/CD

- [ ] **确认 GitHub Actions 构建成功**
  - 访问 `https://github.com/<owner>/<repo>/actions`
  - 检查最新 workflow 运行状态为绿色

- [ ] **确认所有平台构建通过**

## 版本发布详细操作步骤

### 步骤 1：准备工作区

```bash
# 1. 切换到主分支
git checkout main

# 2. 拉取最新代码
git pull origin main

# 3. 确保工作区干净
git status
# 应显示 "nothing to commit, working tree clean"
```

### 步骤 2：运行本地测试

```bash
# 1. 运行单元测试
west build -b native_posix tests/ --build-dir build_tests
west build -t run --build-dir build_tests

# 2. 构建目标板型（验证无编译错误）
west build -t pristine -b nucleo_l4r5zi .

# 3. 检查内存使用
arm-none-eabi-size build/zephyr/zephyr.elf
```

### 步骤 3：更新版本号

```bash
# 1. 查看当前版本
python scripts/bump_version.py
# 输出：1.0.0

# 2. 更新到新版本
python scripts/bump_version.py 1.1.0
# 输出：Version set to 1.1.0 (was 1.0.0)

# 3. 验证变更文件
git status
# 应显示 APP_VERSION、Doxyfile、README.md 被修改
```

### 步骤 4：更新 CHANGELOG

编辑 `CHANGELOG.md`，在顶部添加新版本记录：

```markdown
## [1.1.0] - 2026-04-01

### Added
- 新增板型迁移指南文档
- 添加设备树迁移详细步骤
- 添加 MCU 内存参数速查表

### Changed
- 优化内存配置流程
- 更新文档索引
- 改进 overlay 说明

### Fixed
- 修复小 RAM 板型栈溢出问题
- 修正 overlay 自动加载说明
- 修复文档链接错误

### Deprecated
- 无

### Removed
- 移除冗余的 boards/app.overlay
```

### 步骤 5：提交所有变更

```bash
# 1. 添加所有变更文件
git add APP_VERSION Doxyfile README.md CHANGELOG.md docs/

# 2. 提交变更
git commit -m "chore: 准备发布 v1.1.0

- 更新版本号为 1.1.0
- 更新 CHANGELOG.md
- 更新文档"

# 3. 推送到远程
git push origin main
```

### 步骤 6：等待 CI 验证

- [ ] 访问 GitHub Actions 页面
- [ ] 确认最新 commit 的 workflow 运行成功
- [ ] 确认所有构建任务通过
- [ ] 确认文档生成成功

**如 CI 失败**：
1. 查看失败日志
2. 修复问题
3. 重新提交并推送
4. 等待 CI 再次运行

### 步骤 7：创建 Git 标签

```bash
# 1. 创建带注释的标签
git tag -a v1.1.0 -m "Release v1.1.0

主要更新：
- 新增板型迁移指南
- 优化内存配置流程
- 修复小 RAM 板型问题

详细变更请见 CHANGELOG.md"

# 2. 验证标签
git tag -l
# 应显示 v1.1.0

# 3. 查看标签详情
git show v1.1.0
```

### 步骤 8：推送标签

```bash
# 推送标签到远程
git push origin v1.1.0

# 或推送所有标签
git push origin --tags
```

### 步骤 9：触发 CI 构建（可选）

推送标签后，GitHub Actions 会自动运行 tagged 构建。如未自动运行：

```bash
# 手动触发（如配置了 workflow_dispatch）
# 在 GitHub Actions 页面手动运行
```

## GitHub Release 创建指南

### 方法 1：使用 GitHub Web 界面（推荐）

1. **访问 Releases 页面**
   ```
   https://github.com/<owner>/<repo>/releases
   ```

2. **点击 "Draft a new release"**

3. **填写发布信息**
   - **Tag version**: 选择 `v1.1.0`（刚才推送的标签）
   - **Release title**: `v1.1.0`
   - **Description**: 粘贴 CHANGELOG 内容或自动生成

4. **添加构建产物**（可选）
   - 拖拽上传 `.elf`、`.bin` 文件
   - 上传 API 文档 ZIP

5. **设置发布类型**
   - [x] Set as the latest release（如为正式版）
   - [ ] This is a pre-release（如为预览版）
   - [ ] Create a discussion for this release

6. **点击 "Publish release"**

### 方法 2：使用 GitHub CLI

```bash
# 1. 安装 GitHub CLI（如未安装）
# Windows: winget install GitHub.cli
# macOS: brew install gh
# Linux: sudo apt install gh

# 2. 登录 GitHub
gh auth login

# 3. 创建 Release（自动生成说明）
gh release create v1.1.0 --generate-notes

# 4. 或手动指定标题和说明
gh release create v1.1.0 \
  --title "v1.1.0" \
  --notes-file CHANGELOG.md

# 5. 上传构建产物
gh release upload v1.1.0 \
  build/zephyr/zephyr.elf \
  build/zephyr/zephyr.bin \
  docs/api/html.zip
```

### 方法 3：使用 GitHub API

```bash
# 使用 curl 创建 Release
curl -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  -H "Accept: application/vnd.github.v3+json" \
  https://api.github.com/repos/<owner>/<repo>/releases \
  -d '{
    "tag_name": "v1.1.0",
    "name": "v1.1.0",
    "body": "Release notes...",
    "draft": false,
    "prerelease": false
  }'
```

### 上传构建产物

```bash
# 1. 构建 Release 版本
west build -t pristine -b nucleo_l4r5zi -DCONFIG_DEBUG=n .

# 2. 复制构建产物
mkdir -p release_artifacts
cp build/zephyr/zephyr.elf release_artifacts/
cp build/zephyr/zephyr.bin release_artifacts/

# 3. 打包 API 文档
cd docs/api/html
zip -r ../html.zip .
cd ../../..
cp docs/api/html.zip release_artifacts/

# 4. 上传到 Release（使用 gh CLI）
gh release upload v1.1.0 release_artifacts/*
```

## GitLab Release 创建指南

### 方法 1：使用 GitLab Web 界面

1. **访问 Releases 页面**
   ```
   https://gitlab.com/<owner>/<repo>/-/releases
   ```

2. **点击 "New release"**

3. **填写发布信息**
   - **Tag name**: `v1.1.0`
   - **Release name**: `v1.1.0`
   - **Description**: 粘贴 CHANGELOG 内容

4. **点击 "Create release"**

### 方法 2：使用 GitLab CLI

```bash
# 使用 glab 创建 Release
glab release create v1.1.0 --notes-file CHANGELOG.md
```

### 方法 3：在 `.gitlab-ci.yml` 中自动创建

```yaml
release:
  stage: deploy
  image: registry.gitlab.com/gitlab-org/release-cli:latest
  rules:
    - if: $CI_COMMIT_TAG  # 仅当推送标签时运行
  release:
    tag_name: $CI_COMMIT_TAG
    name: "Release $CI_COMMIT_TAG"
    description: "See CHANGELOG.md for details"
    assets:
      links:
        - name: "Firmware ELF"
          url: "${CI_PROJECT_URL}/-/jobs/${CI_JOB_ID}/artifacts/build/zephyr/zephyr.elf"
        - name: "Firmware BIN"
          url: "${CI_PROJECT_URL}/-/jobs/${CI_JOB_ID}/artifacts/build/zephyr/zephyr.bin"
```

## 附录

### A. 语义化版本规范

```
版本号格式：X.Y.Z

- X (主版本号): 不兼容的 API 变更
- Y (次版本号): 向后兼容的功能新增
- Z (修订号): 向后兼容的问题修正

示例:
1.0.0 -> 1.0.1  # Bug 修复
1.0.0 -> 1.1.0  # 新增功能（向后兼容）
1.0.0 -> 2.0.0  # 破坏性变更
```

### D. 常见问题

**Q1: 版本号更新后构建信息未变？**

A: 清理 CMake 缓存后重新构建：
```bash
west build -t pristine
```

**Q2: Git Dirty 标志始终为 1？**

A: 确保所有更改已提交，或清理生成的文件：
```bash
git clean -fdx
```

**Q3: CI 构建失败？**

A: 查看失败日志，修复问题后重新提交推送。

**Q4: Release 创建后想修改？**

A: 可以编辑 Release 页面修改说明和附件，或删除后重新创建。

**Q5: clang-format 命令找不到？**

A: 需要先安装 clang-format：

```bash
# Windows: 安装 LLVM 或使用 winget
winget install LLVM.LLVM
# 或下载 https://releases.llvm.org/download.html

# 添加 LLVM 到 PATH（如未自动添加）
# C:\Program Files\LLVM\bin

# macOS: 使用 Homebrew
brew install clang-format

# Linux (Ubuntu/Debian):
sudo apt install clang-format

# Linux (Fedora/RHEL):
sudo dnf install clang-tools-extra
```

**Q6: Windows 下 find/xargs 命令报错？**

A: Windows CMD 不支持 Unix 的 find 命令，请使用：
- PowerShell 命令（见上文）
- 或 CMD 的 for 循环命令（见上文）
- 或安装 WSL/Git Bash 使用 Unix 命令
