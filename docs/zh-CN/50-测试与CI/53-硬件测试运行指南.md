# 硬件测试运行指南

本文说明如何将测试用例编译并运行到**真实硬件**上，而非仅在 `native_posix` 模拟器中运行。

---

## 1. 为什么要在硬件上运行测试？

| 测试类型 | 目标平台 | 用途 |
|----------|----------|------|
| **单元测试（native_posix）** | PC（Linux/macOS/WSL） | 快速验证核心逻辑、事件系统、模块管理器等纯软件功能 |
| **硬件测试（真实板卡）** | ARM Cortex-M / RISC-V 等 | 验证外设驱动、中断、定时器、看门狗等硬件相关功能 |
| **集成测试** | 真实板卡 | 验证模块间协作、IPC 通信、多线程并发等系统级功能 |

**推荐流程**：先在 `native_posix` 上跑通单元测试 → 再烧录到硬件验证外设相关功能。

---

## 2. 构建测试固件

### 2.1 选择目标板型

本仓库支持的板型可通过以下命令查看：

```bash
# 列出所有支持的板型
west boards
```

常用板型示例：
- `nucleo_f429zi`（STM32F429）
- `nucleo_f767zi`（STM32F767）
- `disco_l475_iot1`（STM32L475）
- `native_posix`（PC 仿真）

### 2.2 构建测试固件

```bash
# 为目标板构建测试（以 nucleo_f429zi 为例）
west build -b nucleo_f429zi tests/ --build-dir build_tests_hw

# 或使用其他板型
west build -b nucleo_f767zi tests/ --build-dir build_tests_hw
west build -b disco_l475_iot1 tests/ --build-dir build_tests_hw
```

**构建产物**：
- `build_tests_hw/zephyr/zephyr.elf` - ELF 文件（用于调试）
- `build_tests_hw/zephyr/zephyr.bin` - 二进制文件（用于烧录）
- `build_tests_hw/zephyr/zephyr.hex` - Intel HEX 格式

### 2.3 清理构建

```bash
west build -t pristine --build-dir build_tests_hw
```

## 3. 烧录到硬件

### 3.1 使用 west flash（推荐）

```bash
# 烧录到目标板
west flash --build-dir build_tests_hw

# 指定烧录器（如使用 ST-Link）
west flash --build-dir build_tests_hw --runner stlink

# 指定烧录器（如使用 J-Link）
west flash --build-dir build_tests_hw --runner jlink
```

### 3.2 使用 OpenOCD 手动烧录

```bash
# 使用 OpenOCD 烧录 ELF
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg \
  -c "program build_tests_hw/zephyr/zephyr.elf verify reset exit"
```

### 3.3 使用 IDE 烧录

如果你使用 Segger Embedded Studio、Keil、IAR 等 IDE：
1. 导入 `build_tests_hw/zephyr/zephyr.elf`
2. 使用 IDE 的烧录功能将固件下载到目标板

### 5.2 失败输出示例

```
*** Booting Zephyr OS build zephyr-v3.4.0 ***
Running TESTSUITE test_event_system
===================================================================
START - test_event_subscribe
FAIL - test_event_subscribe in 0.002 seconds
Assertion failed at test_event_system.c:85: event_subscribe(...) expected 0 but got -4
...
===================================================================
PROJECT EXEC FAILED
```

### 5.3 常见失败原因

| 错误类型 | 可能原因 | 解决方法 |
|----------|----------|----------|
| `CONFIG_* not defined` | Kconfig 选项未启用 | 检查 `tests/prj.conf` 是否包含所需配置 |
| `Device not found` | 设备树未配置该外设 | 添加板级 overlay 或修改 `tests/prj.conf` |
| `Stack overflow` | 线程栈大小不足 | 增大 `CONFIG_MAIN_STACK_SIZE` |
| `Memory allocation failed` | 堆内存不足 | 增大 `CONFIG_HEAP_MEM_POOL_SIZE` |
| 硬件相关测试失败 | 外设未连接或配置错误 | 检查硬件连接，确认设备树配置正确 |

## 6. 针对硬件的测试配置

### 6.1 修改 tests/prj.conf

`tests/prj.conf` 是测试的默认配置。针对硬件测试，你可能需要调整：

```kconfig
# 增大内存池（硬件通常有更多 RAM）
CONFIG_HEAP_MEM_POOL_SIZE=32768

# 增大主线程栈
CONFIG_MAIN_STACK_SIZE=8192

# 启用硬件看门狗（如果目标板支持）
CONFIG_WATCHDOG=y

# 启用硬件定时器
CONFIG_COUNTER=y

# 启用特定外设驱动（根据测试需求）
CONFIG_GPIO=y
CONFIG_UART=y
CONFIG_I2C=y
CONFIG_SPI=y
```

### 6.2 创建板级专用配置

为不同板型创建专用配置文件：

```bash
# 创建板级配置叠加文件
cp tests/prj.conf tests/prj_nucleo_f429zi.conf

# 编辑特定配置
# 在 tests/prj_nucleo_f429zi.conf 中添加板级特定选项
```

构建时指定配置：

```bash
west build -b nucleo_f429zi tests/ \
  --build-dir build_tests_hw \
  -DCONF_FILE="tests/prj.conf;tests/prj_nucleo_f429zi.conf"
```

### 6.3 设备树覆盖

如果需要修改设备树配置，创建板级 overlay：

```bash
# 创建板级 overlay
cat > boards/nucleo_f429zi.overlay << EOF
&usart2 {
    status = "okay";
    current-speed = <115200>;
};
EOF
```

## 7. 运行特定测试用例

### 7.1 启用/禁用特定测试

在测试文件中，你可以通过条件编译控制哪些测试运行：

```c
/* 仅在硬件上运行的测试 */
#ifdef CONFIG_HARDWARE_TEST
ZTEST(sys_watchdog, test_hardware_wdt) {
    /* 硬件看门狗测试 */
}
#endif

/* 仅在 native_posix 上运行的测试 */
#ifndef CONFIG_HARDWARE_TEST
ZTEST(event_system, test_posix_specific) {
    /* PC 特定测试 */
}
#endif
```

### 7.2 使用 Kconfig 控制测试

在 `tests/Kconfig` 中添加选项：

```kconfig
config HARDWARE_TEST
    bool "Enable hardware-specific tests"
    default n
    depends on GPIO || UART || COUNTER

config TEST_WATCHDOG_HARDWARE
    bool "Test hardware watchdog"
    default y
    depends on HARDWARE_TEST
```

在 `tests/prj.conf` 中启用：

```kconfig
CONFIG_HARDWARE_TEST=y
CONFIG_TEST_WATCHDOG_HARDWARE=y
```

## 8. 调试技巧

### 8.1 使用 GDB 调试

```bash
# 启动 GDB Server（以 ST-Link 为例）
st-util

# 在另一个终端启动 GDB
arm-none-eabi-gdb build_tests_hw/zephyr/zephyr.elf

# GDB 命令
(gdb) target extended-remote :4242
(gdb) break test_event_system.c:85
(gdb) run
(gdb) print subscriber_id
(gdb) continue
```

### 8.2 使用 Segger RTT

```bash
# 启动 JLinkRTTLogger
JLinkRTTLogger -Device STM32F429ZI -RTTChannel 0 -Speed 4000

# 或使用 pyOCD
pyocd rtt --target stm32f429zitx
```

### 8.3 添加调试日志

在测试代码中添加详细日志：

```c
#include <zephyr/logging/log.h>
LOG_MODULE_REGISTER(test_my_module);

ZTEST(my_module, test_something) {
    LOG_INF("Starting test_something");
    
    int ret = my_function();
    LOG_DBG("my_function returned %d", ret);
    
    zassert_equal(ret, 0, "my_function failed");
}
```

## 9. 自动化测试（CI/CD）

### 9.1 在 CI 中运行硬件测试

如果你有 CI 硬件农场（如 Renode、QEMU 或真实板卡），可以在 CI 中添加：

```yaml
# .github/workflows/ci.yml 示例
hardware-tests:
  runs-on: self-hosted
  steps:
    - uses: actions/checkout@v3
    
    - name: Build tests for hardware
      run: |
        west build -b nucleo_f429zi tests/ --build-dir build_tests_hw
    
    - name: Flash and run tests
      run: |
        west flash --build-dir build_tests_hw
        # 使用脚本捕获串口输出并解析结果
        python scripts/capture_test_output.py
```

### 9.2 使用 Twister 运行测试

Zephyr 提供了 `twister` 工具用于自动化测试：

```bash
# 安装 twister
pip install pykwalify

# 运行测试（会自动构建、烧录、收集结果）
twister -T tests/ -p nucleo_f429zi -p nucleo_f767zi

# 生成报告
twister -T tests/ -p nucleo_f429zi --report
```

## 10. 常见问题

### Q1: 编译时报 `undefined reference to` 错误？

**A**: 检查 `tests/CMakeLists.txt` 是否包含了所有需要的源文件。如果测试依赖某些模块，确保它们被添加到 `target_sources` 中。

### Q2: 烧录后板卡无反应？

**A**: 
1. 检查串口连接是否正常
2. 确认波特率配置正确（通常 115200）
3. 检查设备树中 UART 配置是否启用
4. 尝试使用 `west flash --runner stlink` 明确指定烧录器

### Q3: 测试运行但所有用例都失败？

**A**: 
1. 检查 `tests/prj.conf` 是否启用了必要的 Kconfig 选项
2. 查看串口输出中的错误信息
3. 确认硬件连接正确（如 GPIO 引脚、外设电源等）

### Q4: 如何在多个板型上运行同一测试？

**A**: 为每个板型创建单独的构建目录：

```bash
west build -b nucleo_f429zi tests/ --build-dir build_f429
west build -b nucleo_f767zi tests/ --build-dir build_f767
west flash --build-dir build_f429
west flash --build-dir build_f767
```
