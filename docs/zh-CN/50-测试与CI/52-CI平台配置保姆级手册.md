# CI 平台配置保姆级手册（GitHub / GitLab）

本文面向**第一次**在托管平台上为本仓库（或基于本模板的项目）配置**持续集成（CI）的读者：说明本仓库 CI 在做什么、在 GitHub Actions 与 GitLab CI 上如何逐步启用、查看结果、修改版本与板型**，以及**常见问题**。

**相关文档**：[Zephyr版本与CI说明.md](../70-发布与产品化/72-Zephyr版本与CI说明.md) · [单元测试与持续集成说明.md](51-单元测试与持续集成说明.md) · [安全与密钥管理说明.md](../70-发布与产品化/75-安全与密钥管理说明.md)

## 1. 先搞懂：本仓库 CI 在跑什么？

无论使用 GitHub 还是 GitLab，本模板的核心思路一致：


| 阶段                    | 做什么                                                                                                                                                                                             | 依赖                               |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------- |
| **代码质量**              | `shellcheck` 检查 `scripts/*.sh`；`pre-commit run --all-files`（含 **clang-format**、YAML、尾随空白等，见 `**.pre-commit-config.yaml`**）                                                                      | 仅需 Ubuntu + 工具，**不需要** Zephyr 源码 |
| **构建 / 测试**           | 在容器 `**gcr.io/zephyr-project/zephyr-build:v<版本>`** 内执行 `**west init -l .**` → `**west update**` → `**west zephyr-export**`，再 `**west build**`（与本仓库根目录 `**west.yml**` 中 Zephyr 的 `revision` 应对齐） | 需能访问 GitHub（拉 Zephyr）            |
| **文档**                | **Doxygen** 生成 `**docs/api/`**（若存在 `**Doxyfile**`）                                                                                                                                              | 仅需 **doxygen** / **graphviz**    |
| **发布（仅 GitHub 当前配置）** | 打 `**v*`** 标签时收集构建产物并创建 **Release**                                                                                                                                                             | 见下文「GitHub · 标签发布」               |


**版本号一处对齐**：工作流里的 `**ZEPHYR_VERSION`**（如 `3.6.0`）与镜像标签 `**zephyr-build:v3.6.0**`、根目录 `**west.yml**` 里 `**revision: v3.6.0**` 应一致；升级时三处一起改。详见 **[Zephyr版本与CI说明.md](../70-发布与产品化/72-Zephyr版本与CI说明.md)**。

## 2. GitHub Actions（本仓库已自带）

配置文件路径：`**.github/workflows/ci.yml`**。以下按「从零到能跑」的顺序操作。

### 2.1 把代码放到 GitHub 上

1. 在 GitHub 新建一个空仓库（可设为 Private）。
2. 本地进入项目根目录（若尚未初始化 Git）：
  ```bash
   git init
   git remote add origin https://github.com/<你的用户名>/<仓库名>.git
   git add .
   git commit -m "Initial commit"
   git branch -M main
   git push -u origin main
  ```
3. 若你已有远程，只需 `**git remote set-url origin ...**` 后 `**git push**`。

### 2.2 确认 Actions 已启用

1. 打开 GitHub 上该仓库页面。
2. 进入 **Settings → Actions → General**。
3. **Actions permissions** 选择 **Allow all actions and reusable workflows**（或按公司策略选择「仅允许列表内」并放行 `**actions/*`**）。
4. 若组织策略禁用 Actions，需组织管理员放行。

### 2.3 触发第一次流水线

1. 推送任意提交到 `**main**` 或 `**develop**`，或向 `**main**` 发起 **Pull Request**，即会触发 `**.github/workflows/ci.yml`** 中 `on:` 的配置。
2. 打开仓库页签 **Actions**，应看到工作流 **「Zephyr Build CI」** 的运行记录。
3. 点进某次运行，左侧可展开各 **Job**（如 **Code Quality**、**Build (Native POSIX)**、**Unit Tests** 等），点开查看**日志**；红色表示失败，点开失败步骤看末尾报错。

### 2.4 典型失败与自助排查


| 现象                               | 可能原因                   | 处理方向                                                                                                                 |
| -------------------------------- | ---------------------- | -------------------------------------------------------------------------------------------------------------------- |
| **west update 超时 / 无法克隆 Zephyr** | Runner 访问 GitHub 受限    | 换网络、公司代理、或自建 Runner 配置 HTTP(S) 代理；勿在仓库中硬编码 Token。                                                                    |
| **pre-commit / clang-format 失败** | 本地未跑过格式化               | 本地执行 `**pre-commit run --all-files`** 或按日志改对应文件后重推。                                                                  |
| **ShellCheck 失败**                | `scripts/*.sh` 不符合检查规则 | 按日志行号修改，或对确有必要处加 `**# shellcheck disable=...`**（慎用）。                                                                 |
| **west build 某 ARM 板失败**         | 应用配置与该板不兼容             | 改 `**.github/workflows/ci.yml`** 里 `**matrix.board**`，或改 `**prj.conf**` / overlay；与本地 `**west build -b <板名>**` 对齐排查。 |
| **Fork 的仓库 Actions 不跑**          | 默认需本人启用                | Fork 后进入 **Actions** 页，按提示 **I understand my workflows, go ahead and enable them**。                                  |


### 2.5 修改 CI 行为（你要常改的几项）

1. **改 Zephyr 版本**
  - 打开 `**.github/workflows/ci.yml`**，修改顶部 `**env: ZEPHYR_VERSION**`。  
  - 同步修改根目录 `**west.yml**` 中 `**revision:**`（建议 `**vX.Y.Z**` tag）。  
  - 同步 **[Zephyr版本与CI说明.md](../70-发布与产品化/72-Zephyr版本与CI说明.md)** 与 **README** 中的说明（若有硬编码版本）。
2. **改 ARM 冒烟板型**
  - 在 `**build-arm`** 的 `**matrix.board**` 中增删板名；名称必须是 Zephyr 支持的 `**BOARD**`。
3. **改触发分支**
  - 修改 `**on: push: branches`** / `**pull_request: branches**`，例如只保留 `**main**`。
4. **定时任务**
  - `**schedule: cron`** 为每周日运行；不需要可删除 `**schedule**` 段。

### 2.6 标签发布与权限（可选）

工作流末尾 `**release**` 任务在 `**push` 匹配 `refs/tags/v***` 时执行（见 `**if: startsWith(github.ref, 'refs/tags/v')**`）。

1. 本地打标签并推送：
  ```bash
   git tag v1.0.1
   git push origin v1.0.1
  ```
2. **GITHUB_TOKEN** 由 Actions 自动注入，一般**无需**在 Secrets 里配置即可上传 Release 资源。
3. 若使用 `**softprops/action-gh-release`** 等企业策略限制第三方 Action，需在 **Settings → Actions** 中允许该 Action 或换用官方步骤。

### 2.7 Secrets 什么时候要配？

本模板默认 **构建不需要** 自定义 Secret。仅在以下情况在 **Settings → Secrets and variables → Actions** 添加，例如：

- 从**私有** Git 镜像拉 Zephyr（需在脚本里用 Secret 拼 URL，**不要**打印到日志）；
- 发布到非 GitHub 的制品库（需单独 job 与 Token）。

密钥管理原则见 **[安全与密钥管理说明.md](../70-发布与产品化/75-安全与密钥管理说明.md)**。

## 3. GitLab CI（使用仓库内 `.gitlab-ci.yml`）

GitHub 与 GitLab **互斥二选一**也可以**同时**使用：两边各维护一份流水线，**请同步修改** `**ZEPHYR_VERSION`**、板型矩阵与 `**west.yml**`。

### 3.1 准备：把代码推送到 GitLab

1. 在 GitLab（**gitlab.com** 或自建实例）新建项目，可导入 GitHub 仓库或空项目后添加远程：
  ```bash
   git remote add gitlab https://gitlab.com/<组>/<项目>.git
   git push -u gitlab main
  ```
2. 确保仓库根目录存在 `**.gitlab-ci.yml**`（本仓库已提供示例，与 `**.github/workflows/ci.yml**` 能力对齐）。

### 3.2 启用 CI/CD

1. 进入项目 **Settings → General → Visibility, project features, permissions**。
2. 确认 **CI/CD** 功能开启（默认一般已开）。
3. **Settings → CI/CD → General pipelines**：可按需设置 `**Timeout`**、`**Test coverage parsing**` 等，初学者可保持默认。

### 3.3 Runner：共享 Runner 还是自建？

- **GitLab.com 共享 Runner**：免费额度内可直接使用；**无需**自己装机器，推送代码后 **CI/CD → Pipelines** 会出现流水线。  
- **自建 Runner**：在公司内网、需访问私有依赖或拉取 `**gcr.io`** 较慢时，在 **Settings → CI/CD → Runners** 按文档注册 Runner；管理员需保证 Runner 能访问 **GitHub**（`**west update`** 拉 Zephyr）和 **Google Container Registry**（拉 `**zephyr-build`** 镜像）。

若拉 `**gcr.io/zephyr-project/zephyr-build**` 失败：

- 检查 Runner 网络与防火墙；  
- 或在企业镜像站同步该镜像后，把 `**.gitlab-ci.yml**` 里 `**image:**` 改为内网镜像地址。

### 3.4 查看流水线结果

1. 菜单 **CI/CD → Pipelines**：列表中每条流水线有状态（通过 / 失败）。
2. 点进一条 → **Jobs**：与 GitHub 的 Job 对应（如 **code_quality**、**build_native**、**build_tests**、**build_arm** 等）。
3. 失败时点进 Job 看 **日志**；修复后 **git push** 会触发新流水线。

### 3.5 修改 GitLab CI 行为

1. 编辑根目录 `**.gitlab-ci.yml`**。
2. 修改顶部的 `**ZEPHYR_VERSION**` 时，务必同步 `**image: gcr.io/zephyr-project/zephyr-build:v$ZEPHYR_VERSION**`（镜像标签为 `**v` + 版本号**）与 `**west.yml`**。
3. `**build_arm**` 使用 `**parallel: matrix**` 与 `**BOARD**` 列表；改板名方式与 GitHub 的 **matrix.board** 相同。
4. 若某 Job 暂时不需要，可注释掉整段 Job，或加 `**rules:if: '$CI_PIPELINE_SOURCE == "merge_request_event"'`** 等条件（参见 [GitLab CI 文档](https://docs.gitlab.com/ee/ci/yaml/)）。

### 3.6 GitLab 变量与密钥（可选）

路径：**Settings → CI/CD → Variables**。

- `**ZEPHYR_VERSION`** 已在 `**.gitlab-ci.yml**` 的 `**variables:**` 中定义；也可改为在 UI 里覆盖（注意与 `**west.yml**` 一致）。  
- 若将来需要私有依赖 Token，勾选 **Masked**、**Protected**，且**不要**在脚本里 `**echo`** 变量值。

## 4. 双平台对照（维护时一眼对齐）


| 项目            | GitHub                                                              | GitLab                                                    |
| ------------- | ------------------------------------------------------------------- | --------------------------------------------------------- |
| 配置文件          | `**.github/workflows/ci.yml**`                                      | `**.gitlab-ci.yml**`                                      |
| Zephyr 容器     | `**gcr.io/zephyr-project/zephyr-build:v${{ env.ZEPHYR_VERSION }}**` | `**gcr.io/zephyr-project/zephyr-build:v$ZEPHYR_VERSION**` |
| 应用侧 Zephyr 来源 | `**west.yml**` + `**west update**`                                  | 同上                                                        |
| 查看结果          | **Actions** 页                                                       | **CI/CD → Pipelines**                                     |
| 质量门禁          | **Code Quality** job                                                | **code_quality** job                                      |

## 5. 其他平台（Gitee、自建 Jenkins 等）

- **思路相同**：准备含 Zephyr SDK 工具链的环境（或直接使用 `**zephyr-build`** 镜像）、安装 **west**，在流水线中执行与本仓库 **CI 脚本相同**的 `**west init -l .`** → `**west update**` → `**west zephyr-export**` → `**west build ...**`。  
- **Gitee**：若提供类 GitHub Actions 或 Jenkins，可将 `**.github/workflows/ci.yml`** 中的 `**steps**` 翻译为对应语法；注意 Runner 能否访问 **GitHub** 与 **gcr.io**。  
- **不要**把 `**zephyr_config.env`**、私钥提交进仓库；Runner 通过**变量**注入路径类配置。

## 6. 推荐自检（改完 CI 后）

1. 本地 `**west build -b native_posix .`** 与 `**west build -b native_posix tests/**` + `**run**` 通过。
2. `**west.yml**` 中 `**revision**` 与 `**ZEPHYR_VERSION**` / 镜像一致。
3. 推送后平台显示流水线**全部通过**（或已知允许的失败项已在 MR 说明）。
