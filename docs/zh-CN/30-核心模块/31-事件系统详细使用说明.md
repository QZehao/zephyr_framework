## 概述

事件系统是 Zephyr RTOS 框架的核心组件，提供高性能、线程安全的发布 - 订阅（Publish-Subscribe）模式通信机制。该系统支持：

- **异步事件处理**：解耦事件生产者与消费者
- **优先级调度**：支持事件优先级分类
- **线程安全**：所有操作均经过互斥锁保护
- **动态订阅**：运行时动态注册/注销事件类型和订阅者
- **ISR 支持**：支持从中断上下文发布事件

### 主要特性

| 特性 | 描述 |
|------|------|
| 最大事件类型数 | 256 |
| 每类型最大订阅者 | 16 (可配置) |
| 事件队列大小 | 32 (可配置) |
| 调度器优先级 | 5 (可配置) |
| ISR 安全发布 | 支持 |
| **Slab 内存池** | 优先级分层 + 大数据块池 |
| **实时安全 API** | `_rt` 后缀，O(1) 确定分配时间 |
| **分发器** | 支持暂停/恢复、事件过滤、手动批处理 |

## 核心概念

### 事件（Event）

事件是系统中最基本的通信单元，包含以下字段：

```c
typedef struct {
    uint8_t          type;           // 事件类型标识符
    uint8_t          priority;       // 事件优先级
    uint8_t          flags;          // 标志位 (EVENT_FLAG_*)
    uint8_t          reserved;       // 预留扩展
    uint32_t         timestamp;      // 事件创建时间戳（毫秒 uptime）
    uint32_t         source_id;      // 源模块/组件 ID
    uint32_t         data_len;       // 事件数据长度
    union {
        uint8_t  inline_data[CONFIG_EVENT_INLINE_DATA_SIZE]; // 内联数据
        void*    ptr;                                             // 外部数据指针
    } data;
} event_t;
```

**内存布局**（以 64 字节配置为例）：
```
┌────────────────────────────────┐
│ type(1) priority(1) flags(1) ? │  4B
│ timestamp                      │  4B
│ source_id                      │  4B
│ data_len                       │  4B
├────────────────────────────────┤  16B 头部
│ inline_data[48] 或 ptr(8)      │ 48B
└────────────────────────────────┘  64B 总计
```

**数据存储策略**：
- `data_len ≤ CONFIG_EVENT_INLINE_DATA_SIZE`：内联存储，无额外分配
- `data_len > CONFIG_EVENT_INLINE_DATA_SIZE`：从 Slab 池或 k_malloc 分配

**标志位定义**：
| 标志 | 值 | 含义 |
|------|-----|------|
| `EVENT_FLAG_DATA_INLINE` | 0x01 | 数据内联存储于 inline_data |
| `EVENT_FLAG_DATA_DYNAMIC` | 0x02 | 数据动态分配，由系统管理生命周期 |
| `EVENT_FLAG_FROM_SLAB` | 0x04 | event_t 来自 Slab 池 |

### 事件类型（Event Type）

事件类型用于分类事件，系统支持最多 256 种事件类型。预定义的通用事件类型：

```c
#define EVENT_TYPE_GENERIC        1U    // 通用事件
#define EVENT_TYPE_SENSOR_DATA    10U   // 传感器数据事件
#define EVENT_TYPE_SENSOR_CONFIG  11U   // 传感器配置事件
```

### 订阅者（Subscriber）

订阅者是注册接收特定类型事件的回调函数：

```c
typedef struct {
    event_callback_t callback;      // 回调函数
    void            *user_data;     // 用户数据
    uint32_t         subscriber_id; // 唯一订阅者 ID
    bool             is_active;     // 是否激活
} subscriber_entry_t;
```

### 事件优先级

```c
typedef enum {
    EVENT_PRIORITY_LOW      = 10,   // 低优先级
    EVENT_PRIORITY_NORMAL   = 5,    // 普通优先级
    EVENT_PRIORITY_HIGH     = 2,    // 高优先级
    EVENT_PRIORITY_CRITICAL = 0     // 关键优先级
} event_priority_t;
```

### 组件说明

1. **事件队列（Event Queue）**
   - 基于 Zephyr `k_msgq` 实现
   - 支持优先级排队
   - 溢出时丢弃新事件

2. **事件调度器（Event Dispatcher）**
   - 独立线程运行
   - 从队列取出事件并通知订阅者
   - 可配置优先级和栈大小

3. **事件类型管理器（Event Type Manager）**
   - 维护事件类型注册表
   - 管理每个类型的订阅者列表
   - 线程安全的访问控制

## 内存与所有权

本节说明事件在 **入队、分发、回调** 各阶段由谁分配、谁释放，避免误用导致 **use-after-free、双重释放或内存泄漏**。

### Slab 内存管理概述

事件系统支持两种内存分配模式：

| 模式 | API | 内存来源 | 实时安全 | 适用场景 |
|------|-----|----------|----------|----------|
| **Slab 模式** | `event_create_rt`, `event_create_with_data_rt` | 预分配 Slab 池 | ✅ O(1) 确定 | ISR、实时任务 |
| **堆模式** | `event_create`, `event_create_with_data` | k_malloc / Slab 回退 | ⚠️ 可能阻塞 | 普通任务 |

**Slab 池架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                    事件 Slab 池 (event_t)                     │
├───────────────┬───────────────┬───────────────────────────┤
│  CRITICAL 池  │   HIGH 池     │      NORMAL/LOW 池        │
│   (可选)      │   (可选)      │        (必须)             │
├───────────────┴───────────────┴───────────────────────────┤
│                    数据 Slab 池 (可选)                       │
├─────────────┬─────────────┬─────────────────────────────┤
│   256B 池   │    1KB 池   │         4KB 池             │
└─────────────┴─────────────┴─────────────────────────────┘
```

**数据存储策略**：
- `data_len ≤ CONFIG_EVENT_INLINE_DATA_SIZE`（默认 48B）：内联存储，零额外分配
- `data_len ≤ 256`：256B 数据池
- `data_len ≤ 1024`：1KB 数据池
- `data_len ≤ 4096`：4KB 数据池
- `data_len > 4096`：回退到 k_malloc（非实时安全）

### 队列里到底存了什么

事件队列基于 Zephyr `k_msgq` 初始化，单条消息大小为 `sizeof(event_t)`（可配置 32/64/128 字节）：

- `k_msgq_put` 会把调用方传入指针所指的 **`event_t` 整结构体按字节拷贝**到队列环形缓冲区中，**不是**只存「指向 `event_t` 的指针」。
- 结构体里的 `data` 成员是 **联合体**：
  - 小数据（≤ INLINE_DATA_SIZE）：存储在 `inline_data[48]` 中，完整复制
  - 大数据：`ptr` 指针被复制，指向的堆/Slab 内存不复制

分发线程 `k_msgq_get` 时，会把队列槽位里的 `event_t` **再拷贝一份到线程栈上的局部变量**，然后调用 `event_notify_subscribers`，最后在适当时机释放动态分配的数据（见下）。

### `event_publish_copy`（推荐）：模块无需手动释放

流程概要：

1. `event_create_with_data` 分配 **`event_t` 外壳**（优先 Slab，回退 k_malloc）与 **数据缓冲区**（小数据内联，大数据 Slab/k_malloc），并把调用方数据 `memcpy` 进缓冲区。
2. `k_msgq_put` 将 **`event_t` 的值** 写入队列；队列中的副本携带完整事件数据。
3. 入队成功后，实现清除 `EVENT_FLAG_DATA_DYNAMIC` 标志，再 **释放 `event_t` 外壳**（Slab 或 k_free）。
4. 分发线程取出事件、执行回调后，若 `flags & EVENT_FLAG_DATA_DYNAMIC && data.ptr != NULL`，则释放数据内存。

**结论**：走 `event_publish_copy` 时，模块 **不需要** 也 **不应该** 再释放本次事件的 `data` 或 `event_t`；发布失败时由 `event_publish_copy` 内部统一清理。

### `event_publish_copy_rt`（实时安全）：Slab 模式推荐

```c
// ✅ 推荐：ISR 和实时关键任务使用 _rt 后缀 API
void isr_handler(void)
{
    uint8_t data[16] = {0x01, 0x02};
    
    event_status_t ret = event_publish_copy_rt(
        EVENT_TYPE_SENSOR_DATA,
        EVENT_PRIORITY_HIGH,
        data,
        sizeof(data)
    );
    
    if (ret == EVENT_ERR_NO_MEM) {
        // Slab 耗尽，可采取降级措施
    }
}
```

**特点**：
- 完全从 Slab 池分配，O(1) 确定时间
- Slab 耗尽时返回 NULL/错误，**不回退** k_malloc
- 小数据（≤ INLINE_DATA_SIZE）内联存储，零额外分配

### `event_publish`：模块可能仍需负责「外壳」与生命周期

`event_publish` **只负责把当前 `event_t` 拷贝进队列**，**不会** 在发布返回后替你 `k_free` 你通过 `event_create` / `event_create_with_data` 得到的 **`event_t *` 指针本身**。

分发线程在处理完一条消息后 **仅会**：

- 检查 `flags & EVENT_FLAG_DATA_DYNAMIC`，若为真则释放 `data.ptr`

**不会** 释放你当初分配的那颗 `event_t` 结构体。

因此分情形说明：

| 发布方式 | `event_t` 来源 | 数据存储 | 谁释放 `data` | 谁释放 `event_t` 外壳 |
|----------|---------------|----------|---------------|-------------------------|
| `event_publish_copy` | 内部临时 | 内联/Slab/堆 | 分发线程 | 函数内部 |
| `event_publish_copy_rt` | 内部临时 Slab | 内联/Slab | 分发线程 | 函数内部 |
| `event_publish` + 栈上局部 `event_t` | 栈 | 内联 | 无需 | 栈自动结束 |
| `event_publish` + `event_create*` 堆对象 | Slab/堆 | 内联/Slab/堆 | 分发线程 | **须由模块释放** |

**实践建议**：

- **普通任务**：优先使用 `event_publish_copy`
- **ISR/实时任务**：使用 `event_publish_copy_rt`
- **栈上小数据**：可使用 `event_publish` + 栈上 `event_t`（数据内联）

### `event_publish_from_isr`

- ISR 中 **推荐** 使用 `event_publish_copy_rt` 或 `event_create_from_isr` + `event_publish`
- 使用实时安全 API 可保证 O(1) 确定时间，避免堆分配的不可预测性
- **技术细节**：`k_msgq_put(K_NO_WAIT)` 在 ISR 中安全，因为 Zephyr `k_msgq` 内部使用 **spinlock**（自旋锁），而非 mutex（互斥锁）
  - **Mutex**：可能阻塞、触发线程调度，ISR 中禁止使用
  - **Spinlock**：只禁用本地中断、不会调度，ISR 中安全

### 订阅回调中的规则

- 回调收到的 `const event_t *event` 指向 **分发线程栈上的副本**，**仅在回调执行期间有效**
- **不要** 保存该指针供异步使用
- **不要** 在回调里 `k_free(event->data.ptr)`：动态数据由分发线程在回调全部返回后释放

### 与「最佳实践」中示例的对应关系

文档其它处的 **「发布成功后不要访问 `event`」** 主要指：**不要** 再读写已交给 `event_publish` 的那块 **`event_t` 内存的内容**（队列里已是拷贝，改原结构无意义；若仍持有堆指针则涉及泄漏或双重释放策略）。**不等于**「系统会自动 `k_free` 你 `event_create` 返回的指针」——后者仅对 **`event_publish_copy` 内部** 成立。

#### 运行时状态查询（Slab 池）

```c
/**
 * @brief 检查指定优先级的 Slab 是否可用
 * @param priority 事件优先级
 * @return true 有可用块，false Slab 已耗尽或不可用

 * @brief 获取指定优先级 Slab 的剩余块数

 */

/**
 * @brief Slab 统计信息结构体
 */
typedef struct {
    uint32_t critical_used;     // CRITICAL 池已用块数
    uint32_t critical_total;    // CRITICAL 池总块数
    uint32_t high_used;         // HIGH 池已用块数
    uint32_t high_total;        // HIGH 池总块数
    uint32_t normal_used;       // NORMAL 池已用块数
    uint32_t normal_total;      // NORMAL 池总块数
    uint32_t data_256_used;     // 256B 数据池已用块数
    uint32_t data_256_total;    // 256B 数据池总块数
    uint32_t data_1k_used;      // 1KB 数据池已用块数
    uint32_t data_1k_total;     // 1KB 数据池总块数
```

```c
void register_event_types(void)
```

#### 3. 订阅事件

```c
static uint32_t g_subscriber_id;

// 事件回调函数
static void my_event_callback(const event_t *event, void *user_data)
{
    LOG_INF("Received event: type=%d, priority=%d, timestamp=%u",
            event->type, event->priority, event->timestamp);

    if (event->data != NULL && event->data_len > 0) {
        // 处理事件数据
        uint8_t *data = (uint8_t *)event->data;
        // ... 处理数据 ...
    }

    // 访问用户数据
    int *my_data = (int *)user_data;
    LOG_INF("User data: %d", *my_data);
}

void subscribe_to_events(void)
{
    int user_data = 42;

    event_status_t ret = event_subscribe(
        MY_EVENT_TYPE_SENSOR_READ,
        my_event_callback,
        &user_data,  // 用户数据指针
        &g_subscriber_id
    );

    if (ret != EVENT_OK) {
        LOG_ERR("Failed to subscribe: %d", ret);
    }
}
```

#### 4. 发布事件

```c
// 方法 1: 发布静态事件（无动态数据，栈上 event_t）
void publish_simple_event(void)
{
    event_t event = {
        .type = MY_EVENT_TYPE_SENSOR_READ,
        .priority = EVENT_PRIORITY_NORMAL,
        .timestamp = k_uptime_get_32(),
        .source_id = 1,
        .flags = EVENT_FLAG_DATA_INLINE,  // 内联数据，无额外分配
        .data_len = 0
    };

    event_publish(&event);
}

// 方法 2: 带 payload 时优先用 event_publish_copy（推荐）
// 系统会自动管理内存；模块无需 event_free。
void publish_data_event(const void *data, size_t len)
{
    event_publish_copy(
        MY_EVENT_TYPE_DATA_READY,
        EVENT_PRIORITY_HIGH,
        data,
        len
    );
}

// 方法 3: ISR 或实时任务使用 _rt 后缀 API（实时安全）
void isr_publish_event(void)
{
    uint8_t sensor_data[16] = {0x01, 0x02};

    // 完全从 Slab 池分配，O(1) 确定时间
    event_status_t ret = event_publish_copy_rt(
        MY_EVENT_TYPE_SENSOR_READ,
        EVENT_PRIORITY_HIGH,
        sensor_data,
        sizeof(sensor_data)
    );

    if (ret == EVENT_ERR_NO_MEM) {
        // Slab 池耗尽，采取降级措施
    }
}

// 方法 4: 手动创建和发布（不推荐，易泄漏）
void manual_publish(const void *data, size_t len)
{
    event_t *event = event_create_with_data(
        MY_EVENT_TYPE_SENSOR_READ,
        EVENT_PRIORITY_NORMAL,
        data,
        len
    );
    if (event == NULL) {
        return;
    }
    if (event_publish(event) != EVENT_OK) {
        event_free(event);  // 发布失败，释放整颗事件
    }
    // 发布成功时，data 由分发线程释放，但 event_t 外壳不会自动释放
    // 推荐改用 event_publish_copy，避免遗漏
}
```

### 完整示例

```c
#include <zephyr/kernel.h>
#include <zephyr/logging/log.h>
#include "event_system.h"

LOG_MODULE_REGISTER(event_example, LOG_LEVEL_INF);

// 定义事件类型
#define EVENT_TYPE_BUTTON    50U
#define EVENT_TYPE_LED       51U

// 全局订阅者 ID
static uint32_t button_sub_id;
static uint32_t led_sub_id;

// 按钮事件回调
static void button_callback(const event_t *event, void *user_data)
{
    LOG_INF("Button event received: data_len=%u", event->data_len);
    
    if (event->data_len == sizeof(uint8_t)) {
        // 小数据内联存储，直接从 inline_data 读取
        uint8_t button_state;
        if (event->flags & EVENT_FLAG_DATA_INLINE) {
            button_state = event->data.inline_data[0];
        } else {
            button_state = *(uint8_t *)event->data.ptr;
        }
        LOG_INF("Button state: %s", button_state ? "PRESSED" : "RELEASED");
    }
}

// LED 事件回调
static void led_callback(const event_t *event, void *user_data)
{
    LOG_INF("LED event received");
    
    if (event->data_len == sizeof(uint8_t)) {
        uint8_t led_state;
        if (event->flags & EVENT_FLAG_DATA_INLINE) {
            led_state = event->data.inline_data[0];
        } else {
            led_state = *(uint8_t *)event->data.ptr;
        }
        // 控制 LED
        LOG_INF("LED turned %s", led_state ? "ON" : "OFF");
    }
}

// 按钮线程（发布按钮事件）
static void button_thread(void *p1, void *p2, void *p3)
{
    while (1) {
        uint8_t button_state = 1;  // 模拟按钮按下
        
        event_publish_copy(
            EVENT_TYPE_BUTTON,
            EVENT_PRIORITY_NORMAL,
            &button_state,
            sizeof(button_state)
        );

        k_msleep(1000);
        
        button_state = 0;  // 模拟按钮释放
        event_publish_copy(
            EVENT_TYPE_BUTTON,
            EVENT_PRIORITY_NORMAL,
            &button_state,
            sizeof(button_state)
        );

        k_msleep(1000);
    }
}

// LED 控制线程（发布 LED 事件）
static void led_thread(void *p1, void *p2, void *p3)
{
    k_sleep(K_SECONDS(2));  // 等待按钮线程先启动

    while (1) {
        uint8_t led_state = 1;  // LED ON
        
        event_publish_copy(
            EVENT_TYPE_LED,
            EVENT_PRIORITY_LOW,
            &led_state,
            sizeof(led_state)
        );

        k_msleep(500);
        
        led_state = 0;  // LED OFF
        event_publish_copy(
            EVENT_TYPE_LED,
            EVENT_PRIORITY_LOW,
            &led_state,
            sizeof(led_state)
        );

        k_msleep(500);
    }
}

// 主初始化函数
void event_system_example_init(void)
{
    // 1. 初始化事件系统
    event_system_init();

    // 2. 注册事件类型
    event_register_type(EVENT_TYPE_BUTTON, "button");
    event_register_type(EVENT_TYPE_LED, "led");

    // 3. 订阅事件
    event_subscribe(EVENT_TYPE_BUTTON, button_callback, NULL, &button_sub_id);
    event_subscribe(EVENT_TYPE_LED, led_callback, NULL, &led_sub_id);

    // 4. 启动事件调度器
    event_system_start();

    // 5. 创建工作线程
    K_THREAD_DEFINE(button_tid, 1024, button_thread, NULL, NULL, NULL,
                    5, 0, K_NO_WAIT);
    K_THREAD_DEFINE(led_tid, 1024, led_thread, NULL, NULL, NULL,
                    6, 0, K_NO_WAIT);

    LOG_INF("Event system example initialized");
}
```

## 故障排除

### 常见问题

#### 1. 事件未送达

**症状**：发布事件后回调未执行

**检查清单**：
- [ ] 事件类型是否已注册？
- [ ] 订阅者是否成功订阅？
- [ ] 事件调度器是否已启动？
- [ ] 回调函数是否正确实现？

```c
// 调试步骤
void debug_event_delivery(void)
{
    // 检查事件类型
    LOG_INF("Event type name: %s", event_get_type_name(MY_EVENT_TYPE));

    // 检查订阅者数量
    uint32_t count = event_get_subscriber_count(MY_EVENT_TYPE);
    LOG_INF("Subscriber count: %u", count);

    // 检查系统状态
    LOG_INF("System running: %s", 
            event_system_is_running() ? "yes" : "no");
}
```

#### 2. 队列溢出

**症状**：日志显示 "Event queue full, event dropped"

**解决方案**：
```kconfig
# 增加队列大小
CONFIG_EVENT_QUEUE_SIZE=128

# 或优化事件处理速度
# 减少回调中的处理时间
```

#### 3. 内存泄漏 / 野指针

**症状**：系统运行一段时间后内存不足，或随机崩溃

**检查点**：
- 优先使用 `event_publish_copy`；详见 [内存与所有权](#内存与所有权)
- `event_publish` + `event_create*`：分发线程**不会**释放 `event_t` 外壳，堆路径易泄漏；失败路径须 `event_free`
- **禁止** `event_publish` 的 `data` 指向发布函数内的局部栈缓冲（回调可能晚于函数返回）
- ISR 中避免堆分配；`data` 指向的内存在分发前须一直有效
- 回调内不要 `k_free(event->data)`（除非项目另有所有权约定）

#### 4. 死锁问题

**症状**：系统卡死，看门狗复位

**原因**：在事件回调中调用可能阻塞的 API

**解决方案**：
```c
// ❌ 错误示例
static void bad_callback(const event_t *event, void *user_data)
{
    k_mutex_lock(&some_mutex, K_FOREVER);  // 可能死锁
}

// ✅ 正确示例
static void good_callback(const event_t *event, void *user_data)
{
    k_sem_give(&processing_sem);  // 非阻塞
}
```

### 错误码说明

| 错误码 | 值 | 描述 |
|--------|-----|------|
| EVENT_OK | 0 | 成功 |
| EVENT_ERR_NO_MEM | -1 | 内存不足 |
| EVENT_ERR_QUEUE_FULL | -2 | 队列已满 |
| EVENT_ERR_QUEUE_EMPTY | -3 | 队列为空 |
| EVENT_ERR_INVALID_ARG | -4 | 无效参数 |
| EVENT_ERR_NOT_FOUND | -5 | 未找到 |
| EVENT_ERR_NO_SUBSCRIBER | -6 | 无订阅者 |
| EVENT_ERR_TIMEOUT | -7 | 超时 |
