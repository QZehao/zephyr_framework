## 1. 模块化设计入门：是什么？为什么？

### 1.1 什么是模块？

**模块**是软件系统中具有独立职责、可单独测试、可复用的最小单元。在嵌入式 C 语言中，一个模块通常对应：

- 一个 `.c` 源文件 + 一个 `.h` 头文件
- 一组紧密相关的函数和数据结构
- 对外提供清晰定义的接口

**本框架中的模块示例**：

```
src/modules/
├── example_module_gpio.c/h    ← GPIO 驱动模块
├── example_module_uart.c/h    ← UART 通信模块
├── example_module_ipc.c/h     ← IPC 功能模块
```

### 1.2 什么是模块化设计？

模块化设计是一种**将系统分解为独立模块**的软件开发方法，每个模块：
- 封装了特定的职责
- 有明确的接口边界
- 可独立开发、测试、复用

### 1.3 为什么要模块化？

| 不模块化的后果 | 模块化的好处 |
|--------------|-------------|
| 代码重复，复制粘贴满天飞 | 模块复用，减少重复代码 |
| 修改一处，影响一片 | 局部修改，不影响全局 |
| 难以测试，牵一发动全身 | 模块可独立单元测试 |
| 新人上手难，看完全局才敢动 | 模块边界清晰，快速定位 |
| 团队协作冲突，merge 地狱 | 分模块开发，减少冲突 |

### 1.4 模块化设计的核心目标

```
┌─────────────────────────────────────────────────────────────┐
│                    模块化设计的四大目标                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    │
│   │  降低复杂度  │ →  │  提高复用性  │ →  │  便于测试   │    │
│   └─────────────┘    └─────────────┘    └─────────────┘    │
│          │                                    │             │
│          └──────────────→  易于维护 ←─────────┘             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.5 模块化 vs 面向对象

| 维度 | 面向对象 (OOP) | 模块化 (本框架) |
|------|---------------|-----------------|
| 基本单元 | 类 (Class) | 模块 (Module) |
| 封装方式 | 类成员 + 方法 | C 文件 + 接口函数 |
| 继承关系 | 支持多态 | 通过接口模拟 |
| 通信方式 | 方法调用 | 事件发布/订阅 |
| 运行时多态 | 支持 | 编译时确定 |

**本框架的模块化**借鉴了 OOP 的接口概念，但使用 C 语言实现，更适合资源受限的嵌入式环境。

## 2. SOLID 原则深度剖析

### 2.0 SOLID 快速一览

| 原则 | 一句话描述 | 核心问题 |
|------|-----------|---------|
| **S** - 单一职责 | 每个模块只做一件事 | "这个模块会不会因为多个原因改变？" |
| **O** - 开闭原则 | 对扩展开放，对修改关闭 | "加新功能需要改现有代码吗？" |
| **L** - 里氏替换 | 子类可以替换父类 | "所有子类都能互换使用吗？" |
| **I** - 接口隔离 | 客户端只依赖需要的方法 | "模块有没有强迫我调用不需要的接口？" |
| **D** - 依赖反转 | 依赖抽象，不依赖具体 | "高层模块直接调用低层模块了吗？" |

---

### 2.1 单一职责原则 (SRP)

#### 什么是"职责"？

**职责**是模块改变的理由。如果一个模块可以有多个理由被修改，那么它就违反了 SRP。

#### 判断方法：问自己两个问题

```
问题 1：这个模块会因为什么原因改变？
问题 2：有超过一个答案吗？
```

#### 违反 SRP 的例子

```c
// ❌ 违反 SRP：一个模块处理多个职责
// src/modules/bad_module.c
typedef struct {
    uint32_t temperature;      // 可能因为传感器改变
    uint32_t humidity;         // 可能因为传感器改变
    uint32_t display_mode;    // 可能因为 UI 改变
    uint32_t log_level;       // 可能因为配置改变
} sensor_config_t;

int sensor_init(void) {
    // 传感器初始化
}

int sensor_read(uint32_t *temp, uint32_t *hum) {
    // 读取传感器
}

void sensor_display(void) {
    // 显示数据 - UI 职责混入
}

void sensor_log(void) {
    // 记录日志 - 日志职责混入
}
```

**问题分析**：

| 改变原因 | 影响函数 |
|---------|---------|
| 传感器驱动更换 | `sensor_init()`, `sensor_read()` |
| UI 显示逻辑改变 | `sensor_display()` |
| 日志格式改变 | `sensor_log()` |

#### 符合 SRP 的重构

```c
// ✅ 符合 SRP：职责分离
// src/modules/sensor_driver.c/h - 传感器驱动
typedef struct {
    uint32_t temperature;
    uint32_t humidity;
} sensor_data_t;

int sensor_driver_init(void);
int sensor_read_data(sensor_data_t *data);

// src/modules/sensor_display.c/h - 显示模块
void sensor_display_update(const sensor_data_t *data);

// src/app/sensor_logger.c/h - 日志模块
void sensor_log_data(const sensor_data_t *data);
```

#### 本框架中的 SRP 实践

```c
// src/core/event_system.c - 事件系统只做事件分发
int event_publish(event_type_t type, const void *data, size_t len);
int event_subscribe(event_type_t type, event_callback_t cb, void *user_data);
int event_dispatch(const event_t *event);

// 绝不混入：日志功能（应由 sys_log 处理）
// 绝不混入：数据存储（应由其他模块处理）
```

### 2.2 开闭原则 (OCP)

#### 什么是"开"和"闭"？

| 状态 | 含义 |
|------|------|
| **对扩展开放** | 可以新增功能（新增模块、新增事件类型） |
| **对修改关闭** | 现有代码不需要修改 |

#### 违反 OCP 的例子

```c
// ❌ 违反 OCP：每次加新事件都要修改 event_publish
typedef enum {
    EVENT_TYPE_A,
    EVENT_TYPE_B,
    EVENT_TYPE_C,
    // 新增事件需要修改这里
} event_type_t;

int event_publish(event_type_t type, const void *data, size_t len) {
    switch (type) {
        case EVENT_TYPE_A:
            return handle_event_a(data);
        case EVENT_TYPE_B:
            return handle_event_b(data);
        case EVENT_TYPE_C:
            return handle_event_c(data);
        // 每加一个新事件，这里就要改！
        default:
            return -EINVAL;
    }
}
```

#### 符合 OCP 的重构

```c
// ✅ 符合 OCP：新增事件类型无需修改 event_publish
// event_system.c - 核心代码不需要改
int event_publish(event_type_t type, const void *data, size_t len) {
    // 通用的发布逻辑，只操作队列
    return event_queue_push(type, data, len);
}

// src/modules/sensor_module.c - 新增模块无需修改核心
event_subscribe(EVENT_TYPE_SENSOR_DATA, sensor_event_handler, NULL, &sub_id);

// src/modules/network_module.c - 新增模块无需修改核心
event_subscribe(EVENT_TYPE_NETWORK_READY, network_event_handler, NULL, &sub_id);
```

#### 本框架的 OCP 实践

```c
// 核心事件系统对所有事件类型一视同仁
// 新增事件类型 = 新增订阅者 = 无需修改核心

// 定义新事件类型（在 app_config.h）
#define EVENT_TYPE_SENSOR_DATA    10
#define EVENT_TYPE_NETWORK_READY  11

// 新增模块订阅（无需修改 event_system.c）
event_subscribe(EVENT_TYPE_SENSOR_DATA, my_handler, NULL, &my_sub_id);
```

### 2.3 里氏替换原则 (LSP)

#### 什么是里氏替换？

**子类的实例可以替换父类的实例，而不影响程序正确性。**

#### 违反 LSP 的例子

```c
// ❌ 违反 LSP：子类缩小了父类的行为
typedef struct {
    int (*init)(void);
    int (*read)(uint8_t *buf, size_t len);  // 基类：可读任意长度
} sensor_interface_t;

// 错误实现：UART 传感器只能读固定长度
int uart_read(uint8_t *buf, size_t len) {
    if (len != 4) {  // 违反：缩小了 len 的有效范围
        return -EINVAL;
    }
    // ...
}

// 调用方使用父类接口
int generic_read(sensor_interface_t *sensor, uint8_t *buf, size_t len) {
    return sensor->read(buf, len);  // 假设任何 len 都可行
}

// 但传入 UART 传感器时，len != 4 会失败！
```

#### 符合 LSP 的重构

```c
// ✅ 符合 LSP：接口契约明确，所有实现一致
typedef struct {
    const char *name;
    int (*init)(void);
    // read 返回实际读取的字节数，调用方需检查返回值
    int (*read)(uint8_t *buf, size_t len);
} sensor_interface_t;

// UART 实现
int uart_read(uint8_t *buf, size_t len) {
    if (len < 4) {
        return -EINVAL;  // 明确错误码
    }
    // 读取数据
    return 4;  // 返回实际读取字节数
}

// I2C 实现
int i2c_read(uint8_t *buf, size_t len) {
    if (len < 2) {
        return -EINVAL;
    }
    return i2c_transfer(buf, len);
}

// 调用方正确检查返回值
int generic_read(sensor_interface_t *sensor, uint8_t *buf, size_t len) {
    int ret = sensor->read(buf, len);
    if (ret < 0) {
        return ret;
    }
    // 处理读取到的数据
    return 0;
}
```

#### 本框架的 LSP 实践

```c
// 所有模块都实现统一的 module_interface_t
typedef struct {
    const char *name;
    int (*init)(void *config);
    int (*start)(void);
    int (*stop)(void);
    int (*shutdown)(void);
    module_event_handler_t on_event;
    module_status_t (*get_status)(void);
    int (*control)(int cmd, void *arg);
} module_interface_t;

// 模块管理器统一调用，无需关心具体模块类型
int module_manager_register(module_interface_t *interface, void *config, uint32_t *id) {
    // 所有模块都可通过此接口注册
    // 不需要知道具体是 GPIO 模块还是 UART 模块
}
```

### 2.4 接口隔离原则 (ISP)

#### 什么是接口隔离？

**客户端不应依赖它不需要的接口。** 换句话说，不要强迫模块使用它不需要的方法。

#### 违反 ISP 的例子

```c
// ❌ 违反 ISP：巨大臃肿的接口
typedef struct {
    int (*init)(void);
    int (*start)(void);
    int (*stop)(void);
    int (*shutdown)(void);
    int (*read)(uint8_t *buf, size_t len);
    int (*write)(const uint8_t *buf, size_t len);
    int (*flush)(void);
    int (*reset)(void);
    int (*sleep)(void);
    int (*wakeup)(void);
    // ... 更多方法
} mega_driver_interface_t;

// 简单模块也被迫实现所有方法
typedef struct {
    int (*init)(void);
    int (*start)(void);
    int (*stop)(void);
    int (*shutdown)(void);
    int (*read)(uint8_t *buf, size_t len);
    int (*write)(const uint8_t *buf, size_t len);
    int (*flush)(void);      // 简单模块不需要，但必须实现空函数
    int (*reset)(void);      // 简单模块不需要
    int (*sleep)(void);      // 简单模块不需要
    int (*wakeup)(void);     // 简单模块不需要
} simple_uart_interface_t;
```

#### 符合 ISP 的重构：分离接口

```c
// ✅ 符合 ISP：最小化接口，按需组合
// 基础接口（所有模块都需要）
typedef struct {
    int (*init)(void *config);
    int (*start)(void);
    int (*stop)(void);
    int (*shutdown)(void);
} module_base_interface_t;

// 事件处理接口（可选）
typedef struct {
    void (*on_event)(const event_t *event, void *user_data);
} module_event_interface_t;

// 数据读写接口（仅需要读写的模块）
typedef struct {
    int (*read)(uint8_t *buf, size_t len);
    int (*write)(const uint8_t *buf, size_t len);
} module_io_interface_t;

// 简单模块只需实现基础接口
static int simple_init(void *config) { /* ... */ }
static int simple_start(void) { /* ... */ }
static int simple_stop(void) { /* ... */ }
static int simple_shutdown(void) { /* ... */ }

static module_base_interface_t simple_ops = {
    .init = simple_init,
    .start = simple_start,
    .stop = simple_stop,
    .shutdown = simple_shutdown,
};
```

#### 本框架的 ISP 实践

```c
// module_base.h - 核心接口保持最小化
typedef struct {
    const char *name;
    uint32_t version;
    module_priority_t priority;
    const char *const *depends_on;  // 依赖声明
    int (*init)(void *config);
    int (*start)(void);
    int (*stop)(void);
    int (*shutdown)(void);
    module_event_handler_t on_event;  // 事件处理（可选）
    module_status_t (*get_status)(void);
    int (*control)(int cmd, void *arg);  // 控制命令（可选）
} module_interface_t;

// 只需要基础功能的模块，on_event 设为 NULL
// 只需要状态查询的模块，control 可设为 NULL
```

### 2.5 依赖反转原则 (DIP)

#### 什么是依赖反转？

| 传统方式 | 依赖反转后 |
|---------|-----------|
| 高层模块 → 低层模块 | 高层模块 → 抽象接口 ← 低层模块 |

#### 违反 DIP 的例子

```c
// ❌ 违反 DIP：高层模块直接依赖低层模块
// src/app/data_process.c
#include "sensor_driver.h"  // 直接依赖具体实现

int process_sensor_data(void) {
    sensor_data_t data;
    
    // 直接调用低层模块
    if (sensor_driver_read(&data) != 0) {
        return -EIO;
    }
    
    // 处理数据
    return process(&data);
}
```

**依赖关系图**：
```
┌─────────────────┐
│ data_process    │  ← 高层模块
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ sensor_driver   │  ← 低层模块
└─────────────────┘
```

#### 符合 DIP 的重构

```c
// ✅ 符合 DIP：依赖抽象接口
// src/core/sensor_interface.h - 定义抽象接口
#ifndef SENSOR_INTERFACE_H
#define SENSOR_INTERFACE_H

#include <stdint.h>

typedef struct {
    const char *name;
    int (*init)(void);
    int (*read)(uint32_t *data);
    int (*deinit)(void);
} sensor_interface_t;

#endif

// src/app/data_process.c - 依赖抽象
#include "sensor_interface.h"

int process_sensor_data(const sensor_interface_t *sensor) {
    uint32_t data;
    
    // 通过抽象接口调用，不关心具体实现
    if (sensor->read(&data) != 0) {
        return -EIO;
    }
    
    return process(data);
}

// src/drivers/ds18b20.c - 实现接口
#include "sensor_interface.h"

static int ds18b20_read(uint32_t *data) {
    // 读取温度传感器
}

static sensor_interface_t ds18b20_ops = {
    .name = "ds18b20",
    .init = ds18b20_init,
    .read = ds18b20_read,
    .deinit = ds18b20_deinit,
};
```

**依赖关系图**：
```
┌─────────────────┐
│ data_process    │  ← 高层模块
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ sensor_interface│  ← 抽象接口
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌────────┐  ┌────────┐
│ ds18b20│  │ bme280│  ← 低层模块（实现接口）
└────────┘  └────────┘
```

#### 本框架的 DIP 实践

```c
// 高层：事件系统
int event_subscribe(event_type_t type, event_callback_t cb, void *user_data);
// 不依赖具体业务模块，只依赖回调函数签名

// 低层：业务模块实现回调
void my_module_on_event(const event_t *event, void *user_data) {
    // 处理事件，不关心是谁发布的
}

// 通过事件解耦，实现依赖反转
```

## 3. 模块质量评估：内聚度与耦合度

### 3.1 内聚度 (Cohesion) - 模块内部联系有多紧密？

#### 内聚度等级表

| 等级 | 类型 | 说明 | 代码特征 | 推荐程度 |
|------|------|------|---------|---------|
| 7 🟢 | **功能内聚** | 模块只完成单一功能 | 所有代码都服务于一个目的 | ✅✅✅ 最佳 |
| 6 🟢 | **顺序内聚** | 前一输出是后一输入 | 数据处理流水线 | ✅✅ 佳 |
| 5 🟡 | **通信内聚** | 操作相同数据 | 同一数据结构的不同操作 | ✅ 可接受 |
| 4 🟡 | **过程内聚** | 按特定顺序执行 | 步骤明确但功能不同 | ⚠️ 可接受 |
| 3 🔴 | **时间内聚** | 同时执行的操作 | `init()` 放一起 | ❌ 避免 |
| 2 🔴 | **逻辑内聚** | 逻辑相关但功能不同 | `misc()` 函数 | ❌ 避免 |
| 1 🔴 | **偶然内聚** | 无关代码放一起 | 随机组合 | ❌ 禁止 |

#### 内聚度视觉化

```
功能内聚（最佳）          偶然内聚（最差）
┌───────────────┐         ┌───────────────┐
│ ██████████████│         │ ██  █  ██    │
│ ██████████████│         │  █ ██ █  █   │
│ ██████████████│         │ █  ██  ██   │
│ ██████████████│         │  █ ██ █  █   │
└───────────────┘         └───────────────┘
   高度统一                 乱七八糟
```

#### 内聚度判断实战

**场景**：分析以下模块的内聚度

```c
// ❌ 偶然内聚模块
// src/utils/misc.c
int calculate_checksum(const uint8_t *data, size_t len) {
    // 校验和计算
}

void print_debug(const char *fmt, ...) {
    // 调试打印
}

double square_root(double x) {
    // 数学计算
}

int string_utils_split(const char *str, char delim) {
    // 字符串分割
}
// 问题：这些函数之间没有任何关联，只是"杂项"

---

// ✅ 功能内聚模块
// src/utils/crc.c
```

### 3.2 耦合度 (Coupling) - 模块间依赖有多紧密？

#### 耦合度等级表

| 等级 | 类型 | 说明 | 依赖方式 | 推荐程度 |
|------|------|------|---------|---------|
| 1 🟢 | **无耦合** | 完全独立 | 无 | ✅✅✅ 理想 |
| 2 🟢 | **数据耦合** | 通过参数传递 | `func(a, b)` | ✅✅✅ 最佳 |
| 3 🟢 | ** Stamp 耦合** | 通过数据结构 | `func(data_t *d)` | ✅✅ 佳 |
| 4 🟡 | **控制耦合** | 通过标志控制 | `func(cmd, flag)` | ✅ 可接受 |
| 5 🔴 | **外部耦合** | 共享全局数据 | 全局变量 | ❌ 避免 |
| 6 🔴 | **内容耦合** | 直接访问内部 | `a.b = x` | ❌ 禁止 |

#### 耦合度视觉化

```
无耦合（最佳）              内容耦合（最差）
                          
    ┌───┐                  ┌───┐
    │ A │                  │ A │──────┐
    └───┘                  └─┬─┘      │
                              │       │
    ┌───┐                  ┌──▼──┐    │
    │ B │                  │  B  │◄────┘
    └───┘                  └──┬──┘
                              │
    ┌───┐                  ┌──▼──┐
    │ C │                  │  C  │
    └───┘                  └────┘
```

#### 耦合度判断实战

```c
// ✅ 数据耦合：模块通过参数交换数据
// 模块 A
int sensor_get_data(sensor_data_t *out_data) {
    out_data->temp = read_temp();
    out_data->hum = read_hum();
    return 0;
}

// 模块 B 调用 A
sensor_data_t data;
sensor_get_data(&data);  // 只传递数据，不传递行为

// ❌ 外部耦合：共享全局变量
// sensor.c
sensor_data_t g_sensor_data;  // 全局变量

// user.c
extern sensor_data_t g_sensor_data;  // 依赖全局变量
if (g_sensor_data.temp > 30) { ... }

// ❌ 内容耦合：直接访问内部结构
// user.c - 错误示例
extern sensor_data_t g_sensor_data;
g_sensor_data.internal_state = 0xFF;  // 直接修改内部状态
```

#### 本框架耦合度实践

```c
// ✅ 模块间通过事件解耦 - 无直接耦合
// 模块 A 发布事件
event_publish_copy(EVENT_TYPE_DATA_READY, EVENT_PRIORITY_NORMAL, 
                   &sensor_data, sizeof(sensor_data));

// 模块 B 订阅事件

// A 和 B 不知道彼此的存在，只通过事件系统通信
```

### 步骤 1: 识别职责

#### 1.1 什么是职责？

**职责**是模块在系统中承担的**单一、明确的角色或任务**。

#### 1.2 识别职责的方法

**方法一：动词分析法**

从需求中提取动词，动词对应动作，动作对应职责：

```
需求描述：
"系统需要采集温度数据，通过显示模块展示，当温度超过阈值时报警"

动词分析：
- 采集 → 传感器驱动职责
- 展示 → 显示模块职责
- 报警 → 告警模块职责
```

**方法二：变化点分析法**

找出需求中**可能独立变化的部分**：

```
需求中的变化点：
- 传感器类型（DS18B20、BME280、SHT30...）→ 传感器驱动模块
- 显示方式（LCD、OLED、串口...）→ 显示模块
- 告警方式（蜂鸣器、LED、继电器...）→ 告警模块

稳定点（不变的部分）：
- 温度阈值判断逻辑
- 数据采集流程
```

#### 1.3 职责识别检查表

```
□ 这个模块负责什么？
□ 这个模块的名称是否清晰表达其职责？
□ 能否用一句话描述这个模块的职责？
□ 这个模块会因为几个不同的原因改变？
□ 如果这个模块被删除，系统还能正常工作吗？
```

### 步骤 2: 定义接口

#### 2.1 接口设计原则

| 原则 | 说明 | 实践 |
|------|------|------|
| **最小化** | 只暴露必要的方法 | 不需要的方法不暴露 |
| **自描述** | 方法名表达意图 | `sensor_read()` 而非 `sensor_op()` |
| **一致** | 接口风格统一 | 参数顺序、返回值风格一致 |
| **稳定** | 接口一旦发布尽量不变 | 考虑扩展性 |

#### 3.4 循环依赖检测

**循环依赖示例（错误）**：

```c
// ❌ 循环依赖：A → B → C → A
// module_a.c
#include "module_b.h"  // 依赖 B
int a_func(void) {
    b_func();  // 调用 B
}

// module_b.c
#include "module_c.h"  // 依赖 C
int b_func(void) {
    c_func();  // 调用 C
}

// module_c.c
#include "module_a.h"  // 依赖 A - 循环！
int c_func(void) {
    a_func();  // 调用 A
}
```

**解决方案：事件解耦**

```c
// ✅ 解决方案：通过事件解耦，打破循环
// module_a.c - 发布事件，不直接调用
event_publish_copy(EVENT_TYPE_A_DONE, ...);

// module_c.c - 订阅事件，不直接调用
event_subscribe(EVENT_TYPE_A_DONE, c_handler, ...);
```

### 步骤 5: 实现模块

#### 5.1 模块实现模板

```c
/**
 * @file <module_name>.c
 * @brief <模块简短描述>
 *
 * <详细描述，包括设计决策、注意事项等>
 *
 * @author <作者>
 * @version X.Y
 * @date YYYY-MM-DD
 */

#include "<module_name>.h"
#include "app_config.h"

#ifdef CONFIG_<MODULE_NAME_UPPER>
#include <zephyr/kernel.h>
#include <zephyr/logging/log.h>

LOG_MODULE_REGISTER(<module_name>, LOG_LEVEL_DBG);

//==============================================================================
// 私有定义
//==============================================================================

#define MODULE_NAME_MAX_RETRIES  3

static <module_name>_config_t g_config;
static <module_name>_status_t g_status = <MODULE_NAME>_STATUS_NONE;

//==============================================================================
// 私有函数声明
//==============================================================================

static int private_function(int param);
static void event_handler(const event_t *event, void *user_data);

//==============================================================================
// 公共接口实现
//==============================================================================

int <module_name>_init(<module_name>_config_t *config) {
    if (g_status != <MODULE_NAME>_STATUS_NONE) {
        LOG_WRN("Module already initialized");
        return <MODULE_NAME>_OK;
    }

    if (config != NULL) {
        g_config = *config;
    } else {
        // 使用默认配置
        memset(&g_config, 0, sizeof(g_config));
    }

    // 模块特定初始化
    int ret = private_function(0);
    if (ret != 0) {
        LOG_ERR("Init failed: %d", ret);
        g_status = <MODULE_NAME>_STATUS_ERROR;
        return ret;
    }

    g_status = <MODULE_NAME>_STATUS_INIT;
    LOG_INF("Module initialized");
    return <MODULE_NAME>_OK;
}

int <module_name>_start(void) {
    if (g_status == <MODULE_NAME>_STATUS_ERROR) {
        return <MODULE_NAME>_ERR_HW;
    }

    if (g_status == <MODULE_NAME>_STATUS_RUNNING) {
        return <MODULE_NAME>_OK;
    }

    // 启动逻辑
    g_status = <MODULE_NAME>_STATUS_RUNNING;
    LOG_INF("Module started");
    return <MODULE_NAME>_OK;
}

int <module_name>_stop(void) {
    if (g_status == <MODULE_NAME>_STATUS_NONE) {
        return <MODULE_NAME>_OK;
    }

    // 停止逻辑
    g_status = <MODULE_NAME>_STATUS_NONE;
    LOG_INF("Module stopped");
    return <MODULE_NAME>_OK;
}

<module_name>_status_t <module_name>_get_status(void) {
    return g_status;
}
```

## 6. 接口设计详解

### 6.1 好接口的特征

| 特征 | 说明 | 示例 |
|------|------|------|
| **名称自描述** | 看名字知道干什么 | `sensor_read()` vs `sensor_op()` |
| **参数合理** | 参数不多不少 | 读数据只需要 `buf, len` |
| **返回值明确** | 成功/错误清晰 | 0=成功，负数=错误码 |
| **错误处理一致** | 错误码风格统一 | `-EINVAL`, `-ENOMEM` |
| **无副作用** | 调用不改变全局状态 | 不在内部打印日志 |

### 6.2 函数签名设计

#### 6.2.1 输入/输出参数

```c
// ✅ 好：输入参数 const，输出参数指针
int sensor_driver_read(const sensor_driver_config_t *config, 
                       sensor_driver_data_t *data);

// ❌ 差：无法区分输入输出
int sensor_driver_read(sensor_driver_data_t *data);

// ✅ 好：多个相关数据用结构体
int network_send(const network_packet_t *packet);

// ❌ 差：参数列表过长
int network_send(uint8_t *data, size_t len, uint8_t type, 
                uint8_t flags, uint32_t timeout);
```

#### 6.2.2 返回值设计

```c
// ✅ 好：返回错误码，结果通过输出参数返回
int sensor_driver_read(sensor_driver_data_t *data) {
    if (data == NULL) {
        return -EINVAL;
    }
    // 读取数据
    data->temp = read_temp();
    return 0;  // 成功
}

// ❌ 差：返回负数表示错误，但成功时返回数据
int sensor_driver_read(uint32_t *temp) {
    if (temp == NULL) {
        return -EINVAL;
    }
    if (error) {
        return -EIO;
    }
    *temp = actual_temp;
    return actual_temp;  // 混淆：成功时返回温度，失败时返回负数
}
```

## 7. 依赖管理：让你的系统干净整洁

### 7.1 依赖原则

| 原则 | 说明 | 实践 |
|------|------|------|
| **单向依赖** | A → B，不能 B → A | 用事件解耦打破循环 |
| **依赖深度 ≤ 3** | A → B → C → D 太深 | 扁平化设计 |
| **依赖接口** | 依赖抽象，不依赖实现 | 定义接口头文件 |
| **显式声明** | 所有依赖都要声明 | `depends_on` 数组 |

### 7.2 依赖深度控制

```
深依赖（不好）              浅依赖（好）
                          
A → B → C → D              A → B
                          A → C
                          A → D
                          
深度 4                     深度 1-2
```

### 7.3 循环依赖破解三板斧

#### 第一板斧：事件解耦

```c
// ❌ 循环依赖
// module_a.c → module_b.c → module_a.c

// ✅ 破解：事件解耦
// module_a.c 只发布事件
event_publish_copy(EVENT_TYPE_A_REQ, ...);

// module_b.c 只订阅事件
event_subscribe(EVENT_TYPE_A_REQ, b_handle_a_req, ...);

// module_a.c 订阅响应事件
event_subscribe(EVENT_TYPE_B_RESP, a_handle_b_resp, ...);

// module_b.c 发布响应事件
event_publish_copy(EVENT_TYPE_B_RESP, ...);
```

#### 第二板斧：接口抽象

```c
// ❌ 循环依赖
// driver.c → protocol.c → driver.c

// ✅ 破解：提取公共接口
// interface.h - 定义抽象接口
#ifndef INTERFACE_H
#define INTERFACE_H

typedef struct {
    int (*send)(const uint8_t *data, size_t len);
    int (*recv)(uint8_t *buf, size_t len);
} protocol_interface_t;

void interface_register_protocol(protocol_interface_t *iface);

#endif

// driver.c - 实现接口
static int driver_send(const uint8_t *data, size_t len) { ... }
static protocol_interface_t driver_iface = { .send = driver_send, ... };
void driver_init(void) { interface_register_protocol(&driver_iface); }

// protocol.c - 只依赖接口
void protocol_send(const uint8_t *data, size_t len) {
    protocol_interface_t *iface = interface_get_protocol();
    iface->send(data, len);  // 通过接口调用，不直接依赖 driver
}
```

#### 第三板斧：第三层中转

```c
// ❌ 循环依赖
// module_x.c → module_y.c → module_x.c

// ✅ 破解：引入中间层
// translator.c - 中转层
static module_x_interface_t *g_x;
static module_y_interface_t *g_y;

void translator_x_to_y(uint8_t *data) {
    // 转换格式
    convert_x_to_y(data);
    g_y->process(data);
}

void translator_init(module_x_interface_t *x, module_y_interface_t *y) {
    g_x = x;
    g_y = y;
}

// module_x.c 和 module_y.c 都只依赖 translator
```

## 8. 模块生命周期管理

### 8.1 生命周期阶段

```
init ──→ start ──→ run ◄──┐
   │        │              │
   │        ▼              │
   │      stop ────────────┤
   │                       │
   ▼                       │
shutdown ──────────────────┘
```

| 阶段 | 执行时机 | 典型操作 |
|------|---------|---------|
| **init** | 系统启动，模块被注册时 | 分配内存、初始化变量、注册事件 |
| **start** | 模块被用户/系统启动时 | 启动线程、打开硬件、开始工作 |
| **run** | start 后模块持续运行 | 处理事件、响应请求 |
| **stop** | 模块被主动停止时 | 停止线程、关闭硬件、保存状态 |
| **shutdown** | 系统关闭时 | 释放内存、注销事件 |

### 8.2 生命周期实现示例

```c
// 模块生命周期实现示例
int my_module_init(void *config) {
    if (config != NULL) {
        g_config = *(my_module_config_t *)config;
    }
    
    // 分配资源
    g_buffer = k_malloc(BUFFER_SIZE);
    if (g_buffer == NULL) {
        return -ENOMEM;
    }
    
    // 注册事件
    event_subscribe(EVENT_TYPE_DATA, my_module_on_event, NULL, &g_sub_id);
    
    g_status = MODULE_STATUS_INIT;
    return 0;
}

int my_module_start(void) {
    if (g_status != MODULE_STATUS_INIT) {
        return -EINVAL;
    }
    
    // 启动工作线程
    k_thread_start(&g_thread);
    
    g_status = MODULE_STATUS_RUNNING;
    return 0;
}

int my_module_stop(void) {
    if (g_status != MODULE_STATUS_RUNNING) {
        return 0;  // 已经停止
    }
    
    // 停止线程
    k_thread_stop(&g_thread);
    
    g_status = MODULE_STATUS_STOPPED;
    return 0;
}

int my_module_shutdown(void) {
    // 注销事件
    event_unsubscribe(g_sub_id);
    
    // 释放资源
    if (g_buffer != NULL) {
        k_free(g_buffer);
        g_buffer = NULL;
    }
    
    g_status = MODULE_STATUS_NONE;
    return 0;
}
```

## 9. 事件驱动架构：模块间如何通信

### 9.1 事件驱动 vs 直接调用

| 维度 | 直接调用 | 事件驱动 |
|------|---------|---------|
| 耦合度 | 紧密耦合 | 松耦合 |
| 调用关系 | A → B | A → 事件 → B |
| 扩展性 | 差 | 好 |
| 实时性 | 好 | 中（队列延迟） |
| 调试 | 容易 | 困难 |
| 适用场景 | 同步、简单 | 异步、复杂 |

### 9.2 本框架事件通信流程

```
┌─────────┐  发布事件   ┌──────────────┐  分发事件   ┌─────────┐
│ 模块 A  │ ──────────▶ │   事件系统    │ ──────────▶ │ 模块 B  │
│         │             │              │             │         │
│ event_  │             │ event_queue  │             │ on_     │
│ publish │             │              │             │ event() │
└─────────┘             └──────────────┘             └─────────┘
                               │
                               ▼
                        ┌──────────────┐
                        │   事件队列    │
                        │              │
                        │ FIFO + 优先级 │
                        └──────────────┘
```

### 9.4 事件发布/订阅示例

```c
// ========== 发布者（模块 A）==========

// 定义事件数据
typedef struct {
    uint32_t temperature;
    uint32_t humidity;
} sensor_data_event_t;

// 发布事件（带数据拷贝）
sensor_data_event_t data = {
    .temperature = read_temp(),
    .humidity = read_hum(),
};
event_publish_copy(EVENT_TYPE_SENSOR_DATA_READY, EVENT_PRIORITY_NORMAL,
                   &data, sizeof(data));

// ========== 订阅者（模块 B）==========

// 定义处理器
void sensor_display_handler(const event_t *event, void *user_data) {
    if (event->type != EVENT_TYPE_SENSOR_DATA_READY) {
        return;
    }
    
    if (event->len != sizeof(sensor_data_event_t)) {
        return;
    }
    
    const sensor_data_event_t *data = (const sensor_data_event_t *)event->data;
    LOG_INF("Temp: %d, Hum: %d", data->temperature, data->humidity);
}

// 订阅事件（在 init 中）
event_subscribe(EVENT_TYPE_SENSOR_DATA_READY,
                sensor_display_handler,
                NULL,              // user_data
                &g_sensor_sub_id); // 订阅 ID

// 取消订阅（在 shutdown 中）
event_unsubscribe(g_sensor_sub_id);
```

### 9.5 ISR 中的事件发布

```c
// ISR 中使用 event_publish_from_isr
void gpio_isr(const struct device *dev, struct gpio_callback *cb,
               gpio_port_pins_t pins) {
    event_t event = {
        .type = EVENT_TYPE_GPIO_STATE_CHANGED,
        .reserved = 0,
        .data.ptr = &pins,
        .data_len = sizeof(pins),
    };

    // ISR 安全版本（内部使用 spinlock，非 mutex）
    event_publish_from_isr(&event);
}
```

> **注意**：`event_publish_from_isr` 内部调用 `k_msgq_put(K_NO_WAIT)`，该 API 在 Zephyr 中使用 **spinlock** 而非 mutex，因此在 ISR 中是安全的。技术细节见 **[框架核心技术实现细节.md](23-框架核心技术实现细节.md#24-isr-安全发布)**。

## 10. 模块化架构模式与选择

### 10.1 分层架构 (Layered Architecture)

```
┌──────────────────────────────────────────┐
│           表现层 (Presentation)           │
│     Shell 命令、调试接口、状态显示          │
├──────────────────────────────────────────┤
│           应用层 (Application)            │
│     用例协调、模块编排                      │
├──────────────────────────────────────────┤
│            领域层 (Domain)                │
│     业务规则、核心逻辑、事件系统            │
├──────────────────────────────────────────┤
│         基础设施层 (Infrastructure)        │
│     Zephyr 内核、驱动、硬件抽象             │
└──────────────────────────────────────────┘

依赖规则：只能从上往下依赖
```

**适用场景**：大多数嵌入式应用

**本框架实践**：
```
表现层：Shell 命令（app_main.c）
应用层：用例协调
领域层：事件系统、模块管理器
基础设施层：Zephyr 驱动
```

### 10.2 插件架构 (Plugin Architecture)

```
┌─────────────────────────────────────────────┐
│              核心系统 (Core)                 │
│                                              │
│  ┌─────────────────────────────────────┐   │
│  │         模块管理器 (Module Manager)   │   │
│  └─────────────────────────────────────┘   │
│                    │                         │
│     ┌──────────────┼──────────────┐         │
│     ▼              ▼              ▼         │
│  ┌──────┐     ┌──────┐      ┌──────┐     │
│  │ 模块A │     │ 模块B │      │ 模块C │     │
│  │      │     │      │      │      │     │
│  │ 功能1 │     │ 功能2 │      │ 功能3 │     │
│  └──────┘     └──────┘      └──────┘     │
│                                              │
│  特点：模块可插拔、可卸载、可扩展              │
└─────────────────────────────────────────────┘
```

**适用场景**：
- 需要功能裁剪的产品线
- 支持第三方扩展
- 运行时加载模块

### 10.3 事件驱动架构 (Event-Driven Architecture)

```
┌─────────┐  事件  ┌─────────┐  事件  ┌─────────┐
│ 模块 A   │ ──────▶│ 事件总线 │ ──────▶│ 模块 B   │
└─────────┘        └─────────┘        └─────────┘
     │                  │                   │
     │  事件             │  事件             │  事件
     ▼                  ▼                   ▼
┌─────────┐        ┌─────────┐        ┌─────────┐
│ 模块 C   │ ──────▶│ 事件队列 │        │ 模块 D   │
└─────────┘        └─────────┘        └─────────┘
```

**适用场景**：
- 多模块异步协作
- 需要解耦的系统
- 实时性要求高的系统

**本框架实践**：完整实现

## 12. 常见问题与解决方案

### 12.1 上帝模块（God Module）

**症状**：一个模块几千行代码，什么都干

**解决方案**：按职责拆分

```c
// 拆分前
// god_sensor.c (2000 行)
void sensor_init();
void sensor_read();
void sensor_display();
void sensor_log();
void sensor_send_network();
void sensor_handle_button();
void sensor_calibrate();
// ... 2000 行

// 拆分后
// sensor_driver.c - 驱动
// sensor_display.c - 显示
// sensor_network.c - 网络
// sensor_control.c - 控制
```

### 12.2 循环依赖

**症状**：编译报错 `undefined reference` 或初始化顺序问题

**解决方案**：事件解耦或接口抽象

```c
// 方案 1：事件解耦
// A.c - 发布事件，不调用 B
event_publish_copy(EVENT_TYPE_A_REQ, ...);

// B.c - 订阅事件，不调用 A
event_subscribe(EVENT_TYPE_A_REQ, b_handle, ...);

// 方案 2：接口抽象
// a_interface.h - A 提供的接口
// b.c - 实现 A 的接口，注入到 A
```

### 12.3 霰弹式修改（Shotgun Surgery）

**症状**：改一个功能要改 N 个文件

**解决方案**：重新划分模块边界

```c
// 改一个功能需要改：
// - sensor_driver.c
// - sensor_display.c  
// - sensor_network.c
// - sensor_storage.c

// 说明 sensor 模块职责不清，应该拆分：
// - sensor_driver.c - 驱动
// - sensor_data_processor.c - 数据处理
// - sensor_network_sender.c - 网络发送
// - sensor_storage_manager.c - 存储管理
```

### 12.4 深度嵌套

**症状**：函数调用链 A→B→C→D→E→F，5 层以上

**解决方案**：扁平化或依赖注入

```c
// 扁平化前
A() → B() → C() → D() → E() → F()

// 扁平化后
A() → B()    // B 内联或合并
A() → C()    // C 内联或合并
A() → D()    // D 内联或合并

// 或使用依赖注入
A.set_handler(F);  // A 直接调用 F，不通过中间层
```

### 12.5 神秘沉默（Silent Dependencies）

**症状**：模块间隐式依赖，不知道谁依赖谁

**解决方案**：显式声明所有依赖

```c
// 显式声明依赖
static const char *module_a_depends[] = {
    "event_system",
    "sys_log",
    "example_module_b",
    NULL
};

static module_interface_t module_a_interface = {
    .name = "example_module_a",
    .depends_on = module_a_depends,
    // ...
};
```

## 13. 本框架模块化实现指南

### 13.1 创建新模块完整流程

**Step 1：定义模块优先级**（`src/app/app_config.h`）

```c
// 在现有优先级之间添加新优先级
#define APP_INIT_PRIO_MODULE_MINE  60  // 在 MODULE_MGR(54) 和 APP_FINAL(99) 之间
```

**Step 2：创建模块头文件**（`src/modules/my_module.h`）

```c
#ifndef MY_MODULE_H
#define MY_MODULE_H

#include "module_base.h"

typedef struct {
    uint32_t param1;
    uint32_t param2;
} my_module_config_t;

int my_module_init(void *config);
int my_module_start(void);
int my_module_stop(void);
int my_module_shutdown(void);
void my_module_on_event(const event_t *event, void *user_data);
module_status_t my_module_get_status(void);
int my_module_control(int cmd, void *arg);

DECLARE_MODULE_INTERFACE(my_module);

#endif
```

**Step 3：实现模块**（`src/modules/my_module.c`）

```c
#include "my_module.h"
#include "app_config.h"
#include <zephyr/kernel.h>
#include <zephyr/logging/log.h>

LOG_MODULE_REGISTER(my_module, LOG_LEVEL_DBG);

static my_module_config_t g_config;

int my_module_init(void *config) {
    if (config != NULL) {
        g_config = *(my_module_config_t *)config;
    }
    LOG_INF("My module initialized");
    return 0;
}

int my_module_start(void) {
    LOG_INF("My module started");
    return 0;
}

int my_module_stop(void) {
    LOG_INF("My module stopped");
    return 0;
}

int my_module_shutdown(void) {
    LOG_INF("My module shutdown");
    return 0;
}

void my_module_on_event(const event_t *event, void *user_data) {
    // 处理事件
}

module_status_t my_module_get_status(void) {
    return MODULE_STATUS_RUNNING;
}

int my_module_control(int cmd, void *arg) {
    return 0;
}

DECLARE_MODULE_INTERFACE(my_module);

static int my_module_auto_register(void) {
    uint32_t id;
    return module_manager_register(&my_module_interface, &g_config, &id) ? -EIO : 0;
}

SYS_INIT(my_module_auto_register, POST_KERNEL, APP_INIT_PRIO_MODULE_MINE);
```

**Step 4：在 Kconfig 添加配置**（`src/modules/Kconfig`）

```kconfig
config MY_MODULE
    bool "My Module"
    help
      Enable my custom module
```

**Step 5：在 CMakeLists.txt 添加源文件**

```cmake
zephyr_library_sources_ifdef(
    CONFIG_MY_MODULE
    modules/my_module.c
)
```

### 13.2 模块间通信最佳实践

```c
// 1. 定义事件类型（在 app_config.h）
#define EVENT_TYPE_MY_MODULE_DATA  20

// 2. 定义事件数据格式
typedef struct {
    uint32_t value1;
    uint32_t value2;
} my_module_data_event_t;

// 3. 发布事件
my_module_data_event_t data = { .value1 = 1, .value2 = 2 };
event_publish_copy(EVENT_TYPE_MY_MODULE_DATA, EVENT_PRIORITY_NORMAL,
                   &data, sizeof(data));

// 4. 订阅事件（在其他模块）
void other_module_handler(const event_t *event, void *user_data) {
    if (event->type != EVENT_TYPE_MY_MODULE_DATA) {
        return;
    }
    
    const my_module_data_event_t *data = (const my_module_data_event_t *)event->data;
    // 处理数据
}

event_subscribe(EVENT_TYPE_MY_MODULE_DATA, other_module_handler, NULL, &sub_id);
```

## 14. 总结与检查清单

### 14.1 核心原则回顾

```
┌─────────────────────────────────────────────────────────────┐
│                    模块化设计核心原则                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 单一职责（SRP）：每个模块只做一件事                        │
│                                                              │
│  2. 开闭原则（OCP）：对扩展开放，对修改关闭                    │
│                                                              │
│  3. 里氏替换（LSP）：子类可以替换父类                         │
│                                                              │
│  4. 接口隔离（ISP）：客户端不依赖不需要的接口                 │
│                                                              │
│  5. 依赖反转（DIP）：依赖抽象，不依赖具体                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 14.2 模块质量检查表

```
□ 职责单一：一个模块只有一个改变的理由
□ 接口最小化：只暴露必要的公共方法
□ 命名自描述：名称表达意图而非实现
□ 依赖无环：不存在循环依赖
□ 可独立测试：模块可脱离系统测试
□ 错误处理统一：一致的错误码风格
□ 文档完整：公共接口都有注释
□ 内聚度高：功能内聚或顺序内聚
□ 耦合度低：数据耦合或 stamp 耦合
□ 依赖深度浅：≤ 3 层
□ 遵循 SOLID：所有原则都被遵守
```

### 14.3 模块化成功标志

| 标志 | 说明 |
|------|------|
| ✅ 新人能上手 | 看模块头文件就知道怎么用 |
| ✅ 修改局部化 | 改一个功能只改一个模块 |
| ✅ 可独立测试 | 模块不依赖其他模块即可测试 |
| ✅ 无循环依赖 | 依赖图是有向无环图 |
| ✅ 接口稳定 | 改实现不改接口 |
| ✅ 代码可复用 | 模块可在不同项目中复用 |

### 14.4 反模式自查

```
□ 没有上帝模块？（单个模块 < 1000 行）
□ 没有循环依赖？（依赖图无环）
□ 没有霰弹式修改？（改一个功能只改一个模块）
□ 没有深度嵌套？（调用链 < 5 层）
□ 没有隐式依赖？（所有依赖显式声明）
```

```c
/**
 * @file <module_name>.h

 *
 * <详细描述>
 *

#ifndef <MODULE_NAME>_H
#define <MODULE_NAME>_H

extern "C" {
#endif
```

```c
/**
```
