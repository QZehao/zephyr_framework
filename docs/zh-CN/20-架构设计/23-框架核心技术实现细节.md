## 2. 事件系统实现细节

### 2.1 核心数据结构

#### 2.1.1 事件结构体

```c
// event_system.h:177-189
typedef struct {
    uint8_t          type;           // 事件类型 (0-255)
    uint8_t          priority;       // 事件优先级 (0=CRITICAL, 2=HIGH, 5=NORMAL, 10=LOW)
    uint8_t          flags;          // 标志位 (EVENT_FLAG_DATA_INLINE/DYNAMIC/FROM_SLAB)
    uint8_t          reserved;       // 预留扩展
    uint32_t         timestamp;      // 创建时间戳 (毫秒 uptime)
    uint32_t         source_id;      // 源模块 ID
    uint32_t         data_len;       // 数据长度
    union {
        uint8_t  inline_data[CONFIG_EVENT_INLINE_DATA_SIZE]; // 内联数据 (默认 48B)
        void*    ptr;                                             // 外部数据指针
    } data;
} event_t;
```

**内存布局**（以 64 字节配置为例）：
```
┌────────────────────────────────┐
│ type(1) priority(1) flags(1) ? │  4B
│ timestamp                      │  4B
│ source_id                      │  4B
│ data_len                       │  4B
├────────────────────────────────┤  16B 头部
│ inline_data[48] 或 ptr(8)      │ 48B
└────────────────────────────────┘  64B 总计
```

**设计考量**：
- `event_t` 固定大小（由 `CONFIG_EVENT_STRUCT_SIZE` 控制，可配置 32/64/128 字节）
- `data_len ≤ INLINE_DATA_SIZE`（默认 48B）：数据内联存储于 `inline_data`，零额外分配
- `data_len > INLINE_DATA_SIZE`：从 Slab 池或 k_malloc 分配，指针存入 `data.ptr`
- 支持 Slab 模式（`EVENT_FLAG_FROM_SLAB`）实现实时安全分配

### 2.3 队列满处理策略

```c
// event_system.c:480-486
event_status_t event_publish(const event_t* event) {
    // 使用 K_NO_WAIT，不阻塞
    int ret = k_msgq_put(g_event_system.event_queue, event, K_NO_WAIT);
    if (ret != 0) {
        // 原子计数器记录丢弃
        atomic_fetch_add_explicit(&g_event_dropped_count, 1U, memory_order_relaxed);
        LOG_WRN("Event queue full, event dropped (type=%d)", event->type);
        return EVENT_ERR_QUEUE_FULL;
    }
    return EVENT_OK;
}
```

**为什么选择丢弃而非阻塞？**

| 考量因素 | 阻塞等待 | 丢弃事件 |
|----------|----------|----------|
| 实时性 | ❌ 可能导致高优先级线程阻塞 | ✅ 立即返回 |
| 可预测性 | ❌ 延迟不可预测 | ✅ 行为确定 |
| 内存 | 需要更大的队列 | 内存可控 |
| 调用者感知 | ❌ 隐藏压力 | ✅ 显式失败，可降级 |

### 2.4 ISR 安全发布

```c
// event_system.c:497-513
event_status_t event_publish_from_isr(const event_t* event) {
    // 1. 不使用互斥锁（ISR 中禁止）
    // 2. 使用 atomic_get 读取状态（ISR 安全）
    if (atomic_get(&g_event_system.running) == 0) {
        return EVENT_ERR_INVALID_ARG;
    }

    // 3. k_msgq_put 配合 K_NO_WAIT 是 ISR 安全的
    //    内部使用 spinlock（不是 mutex），在 ISR 中允许
    int ret = k_msgq_put(g_event_system.event_queue, event, K_NO_WAIT);
    if (ret != 0) {
        atomic_fetch_add_explicit(&g_event_dropped_count, 1U, memory_order_relaxed);
        return EVENT_ERR_QUEUE_FULL;
    }
    return EVENT_OK;
}
```

#### 关键问题：`k_msgq_put` 内部有锁，为什么 ISR 中可以调用？

**答案：`k_msgq` 使用的是 spinlock，不是 mutex**

| 锁类型 | ISR 中是否允许 | 原因 |
|--------|---------------|------|
| **Mutex** | ❌ 禁止 | 可能阻塞、触发调度，ISR 无法被调度 |
| **Spinlock** | ✅ 允许 | 只禁用本地中断，不会触发调度 |

**Spinlock 在 ISR 中安全的原因：**
```
1. Spinlock 只暂时禁用本地 CPU 中断
2. ISR 本身就在中断上下文，禁用中断不会导致问题
3. 锁持有时间极短（只保护数据结构操作）
4. 不会导致上下文切换或线程调度
```

**Mutex 在 ISR 中危险的原因：**
```
1. Mutex 可能导致线程阻塞
2. 阻塞意味着需要调度其他线程运行
3. ISR 不是线程，无法被"调度走"
4. 尝试阻塞会导致系统崩溃
```

> **Zephyr 官方文档**：
> "This routine is safe to call from an ISR if the timeout is K_NO_WAIT."
> —— [Message Queue API](https://docs.zephyrproject.org/latest/kernel/services/data_passing/message_queues.html)

**ISR 中的限制总结：**

| 操作 | 线程上下文 | ISR 上下文 | 原因 |
|------|-----------|-----------|------|
| Mutex | ✅ 允许 | ❌ 禁止 | 可能阻塞/调度 |
| Spinlock | ✅ 允许 | ✅ 允许 | 只禁用中断 |
| 动态内存分配 | ✅ 允许 | ⚠️ 避免 | 不可重入风险 |
| 阻塞等待 | ✅ 允许 | ❌ 禁止 | ISR 无法调度 |
| 原子操作 | ✅ 允许 | ✅ 允许 | 硬件支持 |
| `k_msgq_put(K_NO_WAIT)` | ✅ 允许 | ✅ 允许 | 内部用 spinlock |

### 2.5 订阅者通知（快照技术）

```c
// event_system.c:727-781
event_status_t event_notify_subscribers(const event_t* event) {
    // 快照结构：保存订阅者的回调和用户数据
    typedef struct {
        event_callback_t cb;
        void*            ud;
    } sub_snap_t;

    sub_snap_t snap[CONFIG_EVENT_MAX_SUBSCRIBERS];
    uint32_t   n = 0U;

    k_mutex_lock(&entry->lock, K_FOREVER);

    // 复制活跃订阅者信息到快照
    for (uint32_t i = 0; i < CONFIG_EVENT_MAX_SUBSCRIBERS; i++) {
        subscriber_entry_t* sub = &entry->subscribers[i];
        if (sub->is_active && sub->callback != NULL) {
            snap[n].cb = sub->callback;
            snap[n].ud = sub->user_data;
            n++;
        }
    }

    k_mutex_unlock(&entry->lock);

    // 在锁外调用所有回调（避免死锁）
    for (uint32_t i = 0; i < n; i++) {
        snap[i].cb(event, snap[i].ud);
    }

    return EVENT_OK;
}
```

**为什么使用快照技术？**

```
问题场景：
  回调函数中可能调用 event_subscribe/unsubscribe
  如果持锁调用回调 → 死锁

解决方案：
  1. 持锁期间只复制回调指针
  2. 释放锁后再调用回调
  3. 回调中可以安全调用任何事件 API
```

### 2.6 事件数据内存管理

事件系统支持三种数据存储策略：

#### 2.6.1 内联数据（小数据，零分配）

```c
// data_len <= CONFIG_EVENT_INLINE_DATA_SIZE (默认 48B)
uint8_t small_data[16];

// 使用 event_publish_copy，内联存储无额外分配
event_publish_copy(EVENT_TYPE_SENSOR_DATA, EVENT_PRIORITY_NORMAL, 
                   small_data, sizeof(small_data));
// 系统自动设置 EVENT_FLAG_DATA_INLINE 标志
```

#### 2.6.2 Slab 池分配（实时安全）

```c
// 完全从 Slab 池分配，O(1) 确定时间，适合 ISR 和实时任务
event_t* event = event_create_with_data_rt(
    EVENT_TYPE_SENSOR_DATA,
    EVENT_PRIORITY_HIGH,
    data,
    data_len
);

// 或直接发布
event_publish_copy_rt(EVENT_TYPE_SENSOR_DATA, EVENT_PRIORITY_HIGH, 
                      data, sizeof(data));

// Slab 池架构：
// - 事件 Slab：CRITICAL/HIGH/NORMAL 按优先级分层
// - 数据 Slab：256B/1KB/4KB 按大小选择
// - 耗尽时返回 NULL，不回退 k_malloc
```

#### 2.6.3 堆分配（普通模式）

```c
// 使用 k_malloc，可能阻塞
event_t* event = event_create_with_data(type, priority, data, data_len);
event_publish(event);
// 注意：需要手动释放 event_t 外壳
if (ret != EVENT_OK) {
    event_free(event);  // 发布失败，手动释放
}
// 或使用便捷函数（自动管理）
event_publish_copy(type, priority, data, data_len);  // 推荐
```

#### 内存管理约定

| 发布方式 | event_t 来源 | 数据存储 | 谁释放 data | 谁释放 event_t |
|----------|-------------|----------|-------------|----------------|
| `event_publish_copy` | 内部临时 | 内联/Slab/堆 | 分发线程 | 函数内部 |
| `event_publish_copy_rt` | Slab | 内联/Slab | 分发线程 | 函数内部 |
| `event_publish` + 栈上 event_t | 栈 | 内联 | 无需 | 栈自动 |
| `event_publish` + `event_create*` | Slab/堆 | 内联/Slab/堆 | 分发线程 | **需模块释放** |

### 2.7 事件队列实现（event_queue）

事件队列基于 Zephyr `k_msgq` 实现，存储完整的 `event_t` 结构体。

```c
// event_queue.h
typedef struct {
    uint32_t enqueue_count;  // 入队操作次数
    uint32_t dequeue_count; // 出队操作次数
    uint32_t overflow_count; // 溢出发生次数
    uint32_t drop_count;    // 事件丢弃次数
    uint32_t high_watermark; // 队列深度历史最大值
} queue_stats_t;
```

**关键特性**：
- 队列满时支持三种溢出策略：`DROP_LOWEST`、`DROP_NEWEST`、`BLOCK`
- 高水位线告警（默认 75% 队列容量）
- 自动释放动态数据载荷（清空队列时）

### 2.8 事件分发器实现（event_dispatcher）

分发器是独立线程，从队列取出事件并通知订阅者。

```c
// event_dispatcher.h
typedef enum {
    DISPATCHER_STOPPED = 0, // 分发器已停止
    DISPATCHER_RUNNING,     // 分发器正在运行
    DISPATCHER_PAUSED,      // 分发器已暂停
    DISPATCHER_ERROR        // 分发器错误状态
} dispatcher_state_t;

typedef struct {
    uint64_t events_processed;  // 已处理的事件总数
    uint64_t events_dropped;    // 被丢弃的事件数
    uint32_t max_latency_us;    // 最大处理延迟（微秒）
    uint32_t avg_latency_us;    // 平均处理延迟（微秒）
    uint32_t processing_errors; // 处理错误次数
} dispatcher_stats_t;
```

**高级功能**：
- **暂停/恢复**：支持动态暂停事件处理（`event_dispatcher_pause/resume`）
- **事件过滤**：可注册过滤函数决定哪些事件被处理
- **手动批处理**：支持手动调用 `process_one` 或 `process_all`
- **延迟统计**：实时监控事件处理延迟

            return -EALREADY;
        }
    }

    module_info_t* info = find_free_slot();

    // 5. 调用 init()（持锁状态下）
    const uint32_t t0 = k_uptime_get_32();
    int ret = interface->init(config);
    const uint32_t elapsed = k_uptime_get_32() - t0;

    // 6. 检查 init 超时
    if (elapsed > CONFIG_MODULE_INIT_TIMEOUT_MS) {
        // 调用 shutdown 清理
        if (interface->shutdown) interface->shutdown();
        return -1;
    }

    // 7. 更新状态
    info->status = MODULE_STATUS_INITIALIZED;
    g_module_mgr.module_count++;

    return 0;

### 4.2 内存分配算法（First-Fit）

```c
// sys_memory.c:355-446
static void* pool_alloc_locked(mem_pool_t* pool, size_t req_size, bool zero, ...) {
    // 1. 对齐请求大小
    size_t aligned_req;
    align_up_size(req_size, MEMORY_ALIGN_BYTES, &aligned_req);

    // 2. 计算总需求（头部 + 数据）
    size_t total_needed = sizeof(alloc_header_t) + aligned_req;

    // 3. First-Fit 查找
    free_block_t* prev = NULL;
    free_block_t* curr = pool->free_list;
    while (curr != NULL) {
        if (curr->size >= total_needed) break;
        prev = curr;
        curr = curr->next;
    }

    if (curr == NULL) {
        pool->fail_count++;
        return NULL;
    }

    // 4. 分割块（如果剩余空间足够大）
    size_t remaining = curr->size - total_needed;
    if (remaining >= MIN_BLOCK_SIZE) {
        // 从当前块尾部切出
        curr->size = remaining;
        block_start = (uint8_t*) curr + remaining;
    } else {
        // 整块分配
        if (prev == NULL) {
            pool->free_list = curr->next;
        } else {
            prev->next = curr->next;
        }
    }

    // 5. 初始化分配头部
    alloc_header_t* header = (alloc_header_t*) block_start;
    header->magic = MEMORY_MAGIC;
    header->pool_type = pool->type;
    header->requested_size = req_size;
    header->aligned_size = aligned_req;

    return block_start + sizeof(alloc_header_t);
}
```

### Q1: 事件队列满时如何处理？

- 原子计数器无需锁，ISR 安全

### Q2: ISR 上下文发布事件有什么特殊处理？

**答案核心：不使用互斥锁 + 原子变量读取状态 + `k_msgq_put` 使用 spinlock**

```c
event_status_t event_publish_from_isr(const event_t* event) {
    // 1. 不使用互斥锁（ISR 禁止）
    // 2. 使用 atomic_get 读取运行状态
    if (atomic_get(&g_event_system.running) == 0) {
        return EVENT_ERR_INVALID_ARG;
    }
    // 3. k_msgq_put 配合 K_NO_WAIT 是 ISR 安全的
    //    内部使用 spinlock（不是 mutex），在 ISR 中允许
    int ret = k_msgq_put(g_event_system.event_queue, event, K_NO_WAIT);
    // ...
}
```

**追问：`k_msgq_put` 内部有锁，为什么 ISR 中可以调用？**

| 锁类型 | ISR 中允许 | 原因 |
|--------|-----------|------|
| **Mutex** | ❌ 禁止 | 可能阻塞、触发线程调度，ISR 无法被调度 |
| **Spinlock** | ✅ 允许 | 只禁用本地中断，不会触发调度 |

**关键区别：**
- Mutex 获取失败时，线程会**阻塞**等待，需要调度器切换到其他线程
- Spinlock 获取失败时，CPU 会**忙等待**（或禁用中断），不会调度
- ISR 不是线程，无法被调度器切换，所以 Mutex 会崩溃，Spinlock 安全

**ISR 限制：**
- ❌ 不能获取 Mutex（会阻塞/调度）
- ⚠️ 避免 `k_malloc`（可能使用 Mutex 或不可重入）
- ❌ 不能使用 `K_FOREVER` 等待（会阻塞）
- ✅ 可以使用 Spinlock（`k_msgq_put(K_NO_WAIT)` 内部用 spinlock）
- ✅ 可以使用原子操作

### Q3: 模块间依赖如何解耦？

**答案核心：字符串声明依赖 + 运行时拓扑排序**

```c
// 编译期声明依赖（不需要 include 其他模块头文件）

**启动时拓扑排序：**

2. 建图：有向边 j→i 表示 i 依赖 j
3. Kahn 算法拓扑排序
4. 检测环，有环则回退到优先级排序

**优点：**

- 支持模块的动态加载和卸载
- 运行时可检测循环依赖

uint32_t sys_mem_check_leaks(sys_mem_pool_type_t type) {
    // 遍历跟踪器，检查魔数
    if (header->magic == MEMORY_MAGIC) {
        leaks++;  // 未释放
    }
}
```

**安全机制：**
| 问题 | 检测方法 |
|------|----------|
| 内存泄漏 | 跟踪器中记录未被移除 |
| 双重释放 | 魔数 `MEMORY_MAGIC` → `MEMORY_FREED_MAGIC` |
| 野指针 | 池范围检查 + 魔数验证 |

### Q5: 为什么使用快照技术通知订阅者？

**答案核心：避免死锁**

```c
// 持锁期间只复制回调指针
k_mutex_lock(&entry->lock, K_FOREVER);
for (uint32_t i = 0; i < CONFIG_EVENT_MAX_SUBSCRIBERS; i++) {
    snap[n].cb = sub->callback;
    snap[n].ud = sub->user_data;
    n++;
}
k_mutex_unlock(&entry->lock);

// 锁外调用回调
for (uint32_t i = 0; i < n; i++) {
    snap[i].cb(event, snap[i].ud);
}
```

**问题场景：**
- 回调函数中可能调用 `event_subscribe/unsubscribe`
- 如果持锁调用回调 → 死锁

**解决方案：**
- 持锁期间只复制回调指针
- 释放锁后再调用回调
- 回调中可以安全调用任何事件 API

### Q6: 内存分配为什么选择 First-Fit？

**答案核心：嵌入式系统特性决定**

| 算法 | 时间复杂度 | 内存利用率 | 碎片程度 |
|------|-----------|-----------|----------|
| First-Fit | O(n) | 中等 | 中等 |
| Best-Fit | O(n) | 高 | 高（小碎片多） |
| Worst-Fit | O(n) | 低 | 低 |

**嵌入式系统特点：**
- 分配请求相对规律（事件、模块数据）
- 更看重分配速度而非极致利用率
- 定期碎片合并可缓解碎片问题

### Q7: 如何保证线程安全？

**答案核心：多层次锁策略**

1. **互斥锁**：保护复杂数据结构
   - 每事件类型独立锁
   - 每内存池独立锁
   - 模块管理器全局锁

2. **原子变量**：单变量读写
   - 事件系统运行状态
   - 事件丢弃计数

3. **快照技术**：避免持锁调用回调
   - 事件通知
   - 模块遍历

4. **锁顺序**：预防死锁
   - 固定锁获取顺序
   - 锁外调用用户代码
