# 参与贡献与代码规范

本文说明如何向本仓库**提交改动**（Issue / Pull Request），以及**代码风格、检查工具与 CI** 的预期，便于贡献者与评审对齐。

**相关**：根目录 **[README.md](../README.md)**「贡献」一节 · **[单元测试与持续集成说明.md](单元测试与持续集成说明.md)** · **[Zephyr版本与CI说明.md](Zephyr版本与CI说明.md)**

---

## 1. 协作流程（建议）

1. **Fork** 本仓库到你的命名空间。  
2. 从 **`main`**（或项目约定的默认分支）**创建功能分支**，命名建议：`feature/简述`、`fix/简述`。  
3. 在分支上完成开发与**自测**（见下文「提交前自检」）。  
4. 发起 **Pull Request（PR）**：标题用**完整句**说明「改了什么、为什么」；正文可附：关联 Issue、测试方式、破坏性变更说明。  
5. 根据评审意见修改；**保持提交历史清晰**（可 rebase / squash，按仓库维护者要求）。

若改动较大，可先开 **Issue** 讨论方向，避免与维护者预期冲突。

---

## 2. 提交前自检

| 项 | 说明 |
|----|------|
| **构建** | 至少 **`west build -b <你的目标板> .`** 通过；若改公共代码，建议再编 **`native_posix`** + **`tests/`**（见 **`tests/README.md`**）。 |
| **格式** | C 代码符合仓库 **`.clang-format`**（见下节）。 |
| **测试** | 与改动相关的 **ztest** 通过；新增逻辑尽量补测试用例。 |
| **文档** | 若改 `Kconfig`、公共 API 或行为，同步更新 **`docs/`**（至少 **[项目配置项说明.md](项目配置项说明.md)** 或专题手册）与必要注释。 |

---

## 3. 代码风格与静态检查

- **格式化**：使用 **clang-format**，风格文件为仓库根目录 **`.clang-format`**。  
  示例：`clang-format -i src/path/to/file.c`  
- **静态分析（可选）**：**`.clang-tidy`** 可在有 **`compile_commands.json`** 时配合 IDE 或命令行使用（参见 **[开发者入门指南.md](开发者入门指南.md)**）。  
  - **提交前钩子（可选）**：若使用 **pre-commit**，配置见 **`.pre-commit-config.yaml`**（安装：`pip install pre-commit && pre-commit install`）。

## 3.5 Commit 提交格式规范

本仓库采用**Conventional Commits**风格的提交格式，确保提交历史清晰、可追溯，便于自动生成变更日志。

### 3.5.1 基本格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

**一行提交示例**：
```bash
git commit -m "feat(sys_memory): 增加堆泄漏检测功能"
```

**多行提交示例**：
```bash
git commit -m "feat(event_system): 优化事件队列性能

- 增加队列溢出统计
- 改进事件丢弃日志
- 添加高水位线监控

Closes: #123"
```

### 3.5.2 `<type>` 类型说明（必填）

| 类型 | 说明 | 何时使用 |
|------|------|----------|
| **`feat`** | 新功能 | 添加新功能、接口、模块 |
| **`fix`** | Bug 修复 | 修复错误行为、逻辑漏洞 |
| **`docs`** | 文档变更 | 修改 README、手册、注释（不改逻辑） |
| **`style`** | 代码格式 | 格式化、空格、缩进（不改逻辑） |
| **`refactor`** | 重构 | 不改功能与 API 的代码结构调整 |
| **`perf`** | 性能优化 | 提升性能、减少内存/CPU 占用 |
| **`test`** | 测试相关 | 添加/修改测试用例、测试配置 |
| **`build`** | 构建系统 | 修改 CMakeLists.txt、Kconfig、CI 配置 |
| **`ci`** | CI/CD | 修改 GitHub Actions、GitLab CI 脚本 |
| **`chore`** | 其他琐事 | 版本号、依赖更新、工具脚本等 |

**不建议的类型**：~~`update`~~、~~`modify`~~、~~`change`~~、~~`apply`~~（过于模糊）

### 3.5.3 `<scope>` 作用范围（可选但推荐）

标注修改的影响范围，帮助快速定位：

| Scope | 说明 | 示例 |
|-------|------|------|
| **`event`** | 事件系统核心 | `event_system`, `event_queue`, `event_dispatcher` |
| **`module`** | 模块管理器 | `module_manager`, 各业务模块 |
| **`service`** | 系统服务 | `sys_log`, `sys_memory`, `sys_timer`, `sys_watchdog` |
| **`ipc`** | Thread IPC 服务 | `ipc_service` |
| **`app`** | 应用主逻辑 | `app_main`, `app_config`, `app_version` |
| **`board`** | 板级支持 | `boards/`, 设备树 overlay |
| **`ci`** | 持续集成 | GitHub Actions, GitLab CI |
| **`docs`** | 文档 | `docs/` 下的手册 |
| **`kconfig`** | Kconfig 配置 | `Kconfig`, `prj.conf`, `prj_*.conf` |

**示例**：
```
feat(event): 增加 EVENT_TYPE_SENSOR_DATA 类型
fix(module): 修复模块启动顺序错误
docs(ci): 更新 CI 板型配置说明
```

### 3.5.4 `<subject>` 标题（必填）

**规则**：
- ✅ 使用**祈使句**现在时（"add" 而非 "added"/"adds"）
- ✅ 首字母**小写**（除非是专有名词）
- ✅ 句末**不加句号**
- ✅ 长度≤50 字符（Git 标准）
- ✅ **中文提交**：使用简洁的中文短语（如「增加堆泄漏检测」而非「增加了堆泄漏检测功能」）

**好的示例**：
```
feat(sys_memory): 增加堆泄漏检测功能
fix(event): 修复事件队列统计错误
docs(app): 更新 app_kv 使用示例
```

**❌ 坏的示例**：
```
修改                          # 过于模糊
修复 bug                      # 未说明范围
添加功能                      # 未说明是什么功能
更新了代码                    # 没有信息量
feat: add new feature         # 没有 scope，不知所云
FIX: 修复严重问题              # 使用大写、感叹号
```

### 3.5.5 `<body>` 正文（可选）

**多行动机说明**（适合复杂改动）：
- 说明**为什么**改动（而非「做了什么」）
- 每行≤72 字符
- 简要交代背景、对比方案、潜在影响

**示例**：
```
Previous event statistics only counted total events.
Adding per-type counters helps identify hot paths and
debug event storms.

Trade-off: increases event_system_cb_t by 4 bytes.
```

### 3.5.6 `<footer>` 页脚（可选）

关联 Issue 或标注破坏性变更：
```
Closes: #123
Closes: #456
References: #789

BREAKING CHANGE: event_subscribe() now requires non-NULL callback.
```

### 3.5.7 常见提交示例

| 场景 | 示例提交信息 |
|------|-------------|
| 添加新功能 | `feat(module): 添加 example_module_gpio 模块` |
| 修复 Bug | `fix(event): 修复事件分发器中断 ISR 竞态` |
| 优化性能 | `perf(sys_memory): 优化 first-fit 算法性能` |
| 文档更新 | `docs(docs): 更新 Thread_IPC 集成指南` |
| 重构代码 | `refactor(event): 简化 event_system_init 流程` |
| 格式化 | `style(src): 使用 clang-format 格式化 event_system.c` |
| 测试用例 | `test(event): 增加事件队列溢出测试` |
| CI 配置 | `ci(github): 增加 nucleo_l4r5zi 构建任务` |

### 3.5.8 Git 配置建议

**设置默认编辑器和自动 trim**：
```bash
git config commit.verbose true
git config grep.lineNumber true
```

**Commit 模板**（可选）：
在项目根目录创建 `.gitmessage`：
```
<type>(<scope>): <subject>


<motivation>
Why this change is needed.

<implementation>
How the problem is solved.

<footer>
Closes: #
```

加载模板：
```bash
git config commit.template .gitmessage
```

## 4. 持续集成（CI）

GitHub Actions 见 **`.github/workflows/ci.yml`**；GitLab 见 **`.gitlab-ci.yml`**。在平台上启用与排查 CI 的步骤见 **[CI平台配置保姆级手册.md](CI平台配置保姆级手册.md)**。流水线通常包含：

- **ShellCheck**：对 **`scripts/*.sh`** 做静态检查。  
- **pre-commit**：与 **`.pre-commit-config.yaml`** 一致（含 **clang-format**、YAML、尾随空白等）；本地可 `pip install pre-commit && pre-commit install` 与 CI 对齐。  
- **构建 / 测试**：**`native_posix`** 主工程与 **`tests/`**、多 ARM 板矩阵冒烟等。

本地 Zephyr 与 CI 镜像版本建议对齐，见 **[Zephyr版本与CI说明.md](Zephyr版本与CI说明.md)**。PR 合并前请确保 CI **绿灯**或说明失败原因（例如仅文档变更触发的已知限制）。

---

## 5. 变更范围与评审友好实践

- **单次 PR 聚焦单一主题**，避免「顺手重构」无关文件，便于评审与回滚。  
- **不提交**本机路径、密钥、大体积二进制（密钥相关见 **[安全与密钥管理说明.md](安全与密钥管理说明.md)**）。  
- 对 **`prj.conf` / `app.overlay`** 的修改，若影响 RAM 或板级行为，请在 PR 描述中说明**目标板**与**验证方式**。
